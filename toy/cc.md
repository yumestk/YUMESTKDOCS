Agent 多步执行时上下文窗口爆炸

  每次工具调用都会在消息列表里塞入大量工具返回结果，积累几轮就逼近 API 的上下文上限。我设计了三层渐进式压缩策略：第一层免费裁剪旧工具输出——只保留头部和尾部，不调LLM，成本为零；第二层把早期消息用 LLM 压缩成一段结构化摘要；第三层硬折叠是逼近上限的最后兜底。

  这里有个比较棘手的技术细节：切分点的选择。如果直接按消息数切，可能把一条 assistant(toolCalls) 和它紧随的 toolResponse 拆开——产生孤立的 tool 消息，OpenAI兼容接口会直接拒绝请求。所以我实现了 safeSplitPoint 算法，整体回溯到安全边界把完整的配对保留在尾部。

  ▎ 💡 钩子 — "为什么分三层而不是直接让 LLM 永远摘要？因为 LLM 调用本身也是 Token
  ▎ 消耗，压缩也要算性价比。第一层免费裁剪本质是工程里最常见的优化思路——优先用最便宜的手段回收资源。那怎么验证三层设计确实比单层好呢？我做了 AgentMetrics埋点，每一层触发次数和回收 token 量都有精确统计" —— 面试官大概率会追问可观测性设计。

模型陷入工具调用死循环

  模型有时会连续多轮请求完全相同的工具+参数组合，不推进任务却持续消耗
  Token。我做了调用签名比对——把工具名和去空白参数合并为签名，连续3轮完全相同就判定为死循环并提前终止。

  ▎ 💡 钩子 — "阈值为什么是3而不是2或者5？这是有说法的——设太低会误杀正常的重试行为，设太高 Token
  ▎ 浪费太多。3是一个平衡点，但说实话这个值本身没有理论最优解。其实更优的方案可能是不计次数而是让另一条模型的判定链路来做仲裁，相当于一个异步看门狗..." ——
  ▎ 引出分布式/多模型审核的思路。

Skill 渐进式披露：不是所有工具都该一次给模型

  早期我给 Agent 一次性塞了十几种工具的完整定义。问题很快就暴露了：工具定义本身就吃掉大量 context token，而且太多工具会让模型的选择准确率下降——这个在学术上叫 tool
  selection degradation。

  我的解决方案是声明式 Skill 系统：每个 Skill 就是一个 YAML frontmatter + Markdown 操作手册的 SKILL.md 文件，放在 classpath 对应的 skills 目录下。Skill声明自己需要哪几个工具子集、system prompt、next step prompt、最大步数。Agent 启动时先只装配 loadSkill + doTerminate两个基础工具，收到用户任务后，从系统提示里的技能目录自主判断匹配哪个 Skill，主动调用 loadSkill 动态注册该 Skill 的工具子集并加载操作手册，然后再开始正式工作。

  ▎ 💡 钩子 — "这个设计借鉴了 Anthropic Claude 官方 Skill 系统的渐进式披露理念。但有个我故意避免的问题：很多框架用反射扫描注解来发现工具，我选择了 Spring 容器里按bean name 做 ToolCallback 映射。为什么？因为反射扫描让 Skill 和具体实现强耦合，但 bean name 映射意味着同一个工具可以有不同的实现，换模型/换存储不需要改 Skill
  ▎ 定义" —— 面试官可能会追着问为什么不用 LangChain 现成的工具管理。

Multi-Agent 并行调度的并发控制与故障隔离

  在深度研报场景里，Team Leader 需要同时派活给 searcher、analyst、writer 三个 Worker。如果串行执行，三个 Worker 各自 60
  秒就是三分钟起步；但如果无限制并发扇出，会瞬间打爆 LLM API 的 RPM 限制。

  我用 JDK 21 虚拟线程 + Semaphore(5) 做了并行调度：Semaphore 控制最大并发 Worker 数，单个 Worker 120 秒超时隔离，单个 Worker 失败不影响其他。

  ▎ 💡 钩子 — "为什么用虚拟线程而不是传统的线程池？Worker 的工作本质是等 LLM 返回、等搜索 API 返回——这是纯 IO 密集场景。虚拟线程在 IO 阻塞时会自动 unmount
  ▎ 出载具线程，所以创建 100 个虚拟线程几乎零开销，而普通线程池 100 个线程每条 1MB 栈就是 100MB 内存起步。但虚拟线程也不是没有坑——如果在 synchronized 块或 JNI
  ▎ 调用里阻塞，它不会自动 unmount，会 pin 住载具线程" —— 面试官大概率会深入追问虚拟线程的原理和坑。

   跨会话的长期记忆丢失

  最早期我用 Kryo 序列化直接把整个消息列表存到本地文件（FileBasedChatMemory），但问题很明显：只能在同一个会话内召回，换一个 conversationId
  就完全找不到历史。而且文件序列化不支持语义检索——比如用户说"之前聊过的那个方案怎么样了"，文件方案完全没有召回能力。

  我后来用 PGVector 替换了这个方案：把每次 agent 的用户查询+助手计划+工具结果构建为向量嵌入，存入独立的 conversation_memory_store 表。下次对话时用当前 query
  做余弦相似度检索，召回 top-K 条历史 exchange，再经轻量 LLM 压缩为一段不超过 200 字的摘要注入到 Agent 的第一步 think 中。整个过程异步执行，不阻塞 Agent
  的主循环。

  ▎ 💡 钩子 — "你可能会问为什么知识库用的是 SimpleVectorStore（内存）但记忆用
  ▎ PGVector？知识库文档量小、不需要持久化、重启重新加载就行；但对话记忆是持续增长的，必须落盘，而且要跨实例共享的话只有 PGVector
  ▎ 这种独立存储能做。另外有个细节——嵌入文本并不是直接把原始对话全量向量化，而是用户 query + 助手计划的前 150 字 + 工具名，因为嵌入模型的 context
  ▎ 有限，全量塞进去反而稀释了语义浓度" —— 面试官可能会追问为什么不用 Chroma/Milvus、嵌入模型的维度和选择。

**完整介绍这个 Multi-Agent 项目整体架构、业务解决什么痛点，ReAct 循环在项目里怎么落地？**

单 Agent 挂载全量工具会造成提示词冗余、模型工具选择出错，因此设计 TeamLeader + 多 Worker 的多智能体协作架构，角色拆分、工具隔离。

*ReAct 循环落地（1min）*
底层基于 think+act 双阶段 ReAct 循环，think 调用大模型决策工具调用，act 落地执行并回填结果；配套四枚举状态机 + 最大执行步数双重兜底规避死循环。核心亮点是嵌套 ReAct：每个 Worker 内部独立运行完整 ReAct 流程，对 Leader 仅表现为一次普通工具调用。

**详细拆解三层渐进式上下文压缩（简历 L1 裁剪→L2 LLM 摘要→L3 硬折叠）整段实现流程，对应上次面经【上下文治理全套连环提问】。**
上下文压缩在每次 think 执行前统一校验，基于 6000token 上限做三段递进压缩，优先低成本方案、逐级升级：
L1（阈值 50%/3000token）：纯字符串裁剪超长历史工具返回，保留调用 ID 保证消息配对，无 LLM 开销，优先回收冗余爬虫 / 搜索原文；裁剪达标就终止后续压缩。
L2（阈值 70%/4200token）：L1 不够时启用，按安全分割点截取前段历史做 LLM 摘要，规避工具调用消息被拦腰拆分导致接口报错；摘要替换原始消息，大模型异常则降级跳过。
L3（阈值 90%/5400token 兜底）：仅留存最近 4 轮消息，剩余全量摘要折叠；LLM 调用失败直接填充占位文本，从根源杜绝上下文超限报错。

1. 细节深挖（Q3~Q6，抠落地参数、阈值、选型）
**token 压缩触发阈值是多少？**
三个阈值都基于 6000 token 上限按比例触发：
     
  ┌─────────────┬───────────────────────────────┬──────────────────────────────────┐
  │    层级      │           触发阈值              │               机制               │
  ├─────────────┼───────────────────────────────┼──────────────────────────────────┤
  │ L1 免费裁剪  │ token 超过 3000（上限的 50%）     │ 裁剪旧的大体积工具输出，不调 LLM │
  ├─────────────┼───────────────────────────────┼──────────────────────────────────┤
  │ L2 LLM 摘要 │ token 超过 4200（上限的 70%）    │ 把前 40% 的旧消息送 LLM 压成摘要 │
  ├─────────────┼───────────────────────────────┼──────────────────────────────────┤
  │ L3 硬折叠    │ token 超过 5400（上限的 90%）   │ 只保留最近 4 条 + 摘要，其余全砍 │
  └─────────────┴───────────────────────────────┴──────────────────────────────────┘

  另外有两个前置条件：消息少于 6 条不触发任何压缩；L2 还要求至少 10 条消息才执行，避免消息太少时把有效上下文也压没了

**用什么工具预估 token（JTokkit），为什么不用 tiktoken？**（对标上次 Q3 token 超限压缩阈值）
选 JTokkit 不是因为它比 tiktoken 好，是因为 tiktoken 根本没法用——它是 Python 库，我整个项目是 Java 技术栈。在 JVM 生态里，JTokkit 是 tiktoken的直接移植，用的同一套 BPE 算法、同一个 cl100k_base 编码表，计出来的 token 数跟 OpenAI 官方 tiktoken 是一致的。
     
那为什么不用字符数除以四这种简单估算？因为中英文混着用时这个误差太大了。一个英文字母在 cl100k_base 里大概 0.5 个 token，但一个中文字一般是 1 到 3 个 token，你拿chars/2 算只能碰运气，chars/4 更离谱。我的上下文窗口上限设的 6000 token，如果估算偏差了 30%，等于实际触发压缩的时机晚了 1800 token——上下文快爆了才动手，API调用就可能被拒绝。压缩系统的阈值判断依赖准确计数，你连自己还剩多少空间都不知道，怎么决定什么时候压缩？
所以用了 JTokkit，编码选 CL100K_BASE。有个细节是初始化用了懒加载——newLazyEncodingRegistry() 不会在类加载器阶段就解压整个编码表，只在第一次 count调用时才加载，启动不拖慢。另外做了一个 fail-safe 兜底——如果 JTokkit 任何原因抛了异常，回退到 chars/2，保证 token 计数这根链路永远不会中断 Agent 的主执行流程。


**L1、L2、L3 三层各自压缩规则是什么？**
三层逐级压缩，触发阈值、留存策略各不相同：
L1（3000token）：保留最新 6 条消息，超长工具返回首尾截存、保留 ID 保证配对，纯字符串处理无 LLM，达标即终止后续压缩。
L2（4200token）：原计划切前 40% 历史摘要，通过安全切点规避工具消息被拆分，失败则直接跳过。
L3（5400token 兜底）：仅留存最后 4 条，剩余全部摘要；LLM 异常改用固定占位文本，避免超限报错。

**L1 优先回收 tool_call/tool_response 的设计原因？为什么避开配对**
两个原因：一个是直接收益——tool response 就是上下文膨胀的头号元凶；另一个是硬约束——如果拆了配对 API 直接报错。
先说收益。Agent 跑起来以后，searchWeb 可能返回几百行的搜索结果、scrapeWebPage 抓回来一整个网页的 HTML、readFile
读一个大文件——这些数据模型看过一次就够了。后面几步的推理不需要"百度搜索结果的第三页第七行是啥"，它只需要知道"搜过了，搜到了什么方向的结果"。所以 L1 优先回收tool response 是在捡最肥的肉吃——改动最小、回收的 token 空间最多。
再说为什么必须保留 id 和 name、必须避开切断配对。在 OpenAI 兼容的 chat completion API 里，消息列表中的 assistant(toolCalls) 和 toolResponse 是严格配对的。每一条assistant 消息里的 tool_calls 数组声明了"我要调用这几个工具，每个工具的 id 是 xxx"，后面必须紧跟对应 id 的 tool 消息。如果我把 toolResponse 的 id裁掉了，或者把 assistant 留在压缩侧、toolResponse 留在保留侧——API 收到请求后发现有个 tool_call 的 id 找不到对应的 tool 消息回应，直接返回 400 错误。这就是 safeSplitPoint 的价值。它不是简单的"切前40%"，而是扫一遍切分点附近的相邻消息，发现有拆散配对的风险就整体平移，宁可少压缩一点也不能把合法消息序列变成非法请求。

**L2具体怎么做的内容提炼压缩？**（对标上次 Q4 压缩实现细节）
L2 的压缩分四个环节——筛选范围、安全切分、格式化投喂、注入回写。
  *筛选范围*
  消息分两段：前 40% 是"旧消息"要被压缩，后 60% 保留不动。但这个 40% 是期望值，实际切分点要过 safeSplitPoint 调整。
  *格式化投喂*
  旧消息不能直接灌给 LLM。每条消息先取 getText() 拿纯文本，截前 200 字符，太长就加 ...。然后标上序号和角色类型——[1] [USER] xxx, [2] [ASSISTANT] xxx, [3] [TOOL] xxx。空消息直接跳过。最后拼接成一个编号列表字符串，塞进压缩提示词模板里。
  提示词模板要求三件事：保留关键信息（已完成的任务步骤、重要发现、收集到的数据、当前进度、待完成事项），忽略重复内容和冗余的工具返回数据，输出不超过 300字的中文摘要。
  *LLM 调用*
  用的是 Agent 自己的 chatClient，也就是同一个模型实例，不走额外的模型配置。调一次同步 call，拿到摘要文本。
  *注入回写*
  摘要生成后，旧消息全部从列表里删掉，在列表开头插入一条 [推理历史摘要]\n + 摘要内容的 UserMessage。这条新消息在后续的 think 循环里会被正常读入上下文。
  *降级兜底*
  LLM 调用失败就静默跳过——try-catch 包住，warn 日志打一条，返回 false 表示没执行压缩。L3 更激进，LLM 失败时直接用占位文本 (早期上下文已折叠) 顶上，因为到 L3已经在 90% 阈值了，不能再等了

**三层上下文治理最终落地输出是什么？是全量拼接进 prompt 还是分层存储？**（对标上次 Q5 治理结果）
落地输出的形态很直接：三层作用于同一个 messageList，最终产出物是一个扁平列表，全量拼接进下一次 LLM 调用的 prompt。没有分层存储，没有独立缓存层。
  压缩前消息列表可能是这样：
  [0] UserMessage       "研究 2024 年 AI Agent 融资情况"
  [1] AssistantMessage   think: 需要搜索...
  [2] ToolResponse       searchWeb 返回 3000 字...
  [3] AssistantMessage   think: 还需要搜百度...
  [4] ToolResponse       searchWeb 返回 2500 字...
  [5] AssistantMessage   think: 搜阿里...
  [6] ToolResponse       searchWeb 返回 2800 字...
  [7] AssistantMessage   think: 搜字节...
  [8] ToolResponse       searchWeb 返回 3200 字...  ← L1 触发，这里被裁剪
  [9] AssistantMessage   think: 需要验证...
  [10] ToolResponse      searchKnowledgeBase 返回 1500 字...
  [11] AssistantMessage  think: 写报告...
  [12] ToolResponse      doTerminate 返回...

  L2 执行后变成：
  [0] UserMessage       "[推理历史摘要] 已完成百度、阿里、字节等多家公司的搜索..."
  [1] AssistantMessage   think: 搜字节...                    ← 后 60% 保留
  [2] ToolResponse       searchWeb 返回 3200 字（已被 L1 裁剪）
  [3] AssistantMessage   think: 需要验证...
  [4] ToolResponse       searchKnowledgeBase 返回 1500 字...
  [5] AssistantMessage   think: 写报告...
  [6] ToolResponse       doTerminate 返回...
  就是一个扁平列表。下一次 think() 调 new Prompt(messageList, chatOptions) 时，这些消息——摘要在前，最近消息在后——全部拼进 prompt，没有任何分层存储的概念。

  *为什么是扁平而非分层？*
  最根本的原因是 LLM API 不认"层"这个概念。你发给 OpenAI 或 DashScope 的 messages
  数组就是一维列表，没法标注"这条是摘要层、这条是原始消息层"。如果你想做分层，只能自己在应用层维护多个 bucket，每次拼 prompt
  时按规则从各层取数据再组装。但这样增加了复杂度，收益在哪里？摘要已经压缩为 300 字了，跟原始消息放在一起并不会显著挤占窗口——它只占 300 token，而后 60%的保留消息可能有 2000 多 token。分桶维护的复杂度不值得为这点空间优化。
  *但分层思想有没有体现？*
  体现在触发层级的递进决策上，而不是存储形态上。L1 不动消息结构、L2 把前段替换为摘要、L3 切到只剩 4条——三层是在同一个列表上施加渐进式破坏，破坏程度递增但始终维持一个扁平列表。另外，L3 跨会话长期记忆那套（ConversationMemoryService）走的是 PGVector独立存储、语义检索、LLM 摘要后再注入——那个是真正的外部分层，但它跟 L1/L2/L3 不在同一个存储域上：L1/L2/L3 管的是当前会话内的 messageList 消瘦化，PGVector管的是跨会话的历史记忆回流，两条线各司其职，不混在一起。

**跨会话长期记忆基于 PGVector+LLM 摘要召回：什么时候触发历史召回？召回多少条？全部塞进 prompt 吗？**
我的长期记忆方案是基于 PGVector 检索 + LLM 摘要，触发时机、召回数量和注入策略是这样的：
*触发时机*：只在每个新会话的第一轮对话触发一次，用一个一次性标志位保证整个会话生命周期内只召回一次，避免重复注入浪费 token。
*召回数量：取与当前问题最相似的* 3 条历史记录。太少覆盖不足，太多摘要质量下降且挤占上下文，3 条是在信息覆盖和精度之间的平衡。
*注入策略*：不是把原始记录全塞进去，而是先用 LLM 把这 3 条历史压缩成 200 字以内的摘要，然后把摘要作为第一条消息注入到对话开头。这样既让模型知道历史信息，又不会占用过多上下文窗口。如果摘要失败，还有降级截断兜底。

3. 字节灵魂抬杠反问（重点，对标上次 Q6~Q8 连环找茬，面试官高频挖坑）
**假设现在连续对话 12 轮：前 10 轮经过 L1/L2/L3 压缩落长期记忆，第 11、12 轮新消息未压缩，你入参 LLM 是【压缩历史 + 最新两条消息】吗？**
先纠正一个理解：两套系统不串在同一条链路上。
  会话内 L1/L2/L3 作用于当前 run() 的 messageList，管的是"这一次 Agent 执行过程中消息太多怎么办"。跨会话长期记忆（ConversationMemoryService）作用于PGVector，管的是"下一次启动 Agent 时如何把前几次对话的相关经验带过来"。它们的触发时机完全不同：
    L1/L2/L3：在当前执行的每一步 think() 开头检查并压缩
    长期记忆注入：在新会话的第一次 think() 时从 PGVector 检索并注入
  ---
  那具体到你说的第 13 轮新会话，入参 LLM 的形态是什么？
  
  启动新会话时，YumeManus.think() 在第一次调用 super.think() 之前，先走 memoryService.retrieveAndSummarize(originalUserPrompt, 3)，从 PGVector
  里语义检索跟当前问题最相关的历史 exchange，用 LLM 压缩为 ≤200 字的摘要，插入到消息列表的第 0 位：

  [0] UserMessage  "[历史对话记忆]\n之前你研究过AI Agent融资，搜索了百度阿里字节，结论是..."  ← 跨会话摘要
  [1] UserMessage  "帮我更新一下上次的研报数据"                                      ← 当前用户输入

  然后随着 Agent 开始执行，消息列表逐步增长。假如跑到第 8 步触发了 L2，列表变成：

  [0] UserMessage  "[历史对话记忆]\n..."                                            ← 跨会话摘要（不动）
  [1] UserMessage  "[推理历史摘要]\n已完成搜索百度阿里，发现新融资数据..."              ← L2 会话内摘要
  [2] AssistantMessage think: 搜字节...                                             ← 最近 60% 保留
  [3] ToolResponse ...                                                              ← 完整保留
  [4] AssistantMessage think: 写报告...
  [5] ToolResponse ...

  跨会话摘要和会话内摘要同时在列表里，但来源和职责不同。

  ---
**那新消息没压缩，靠压缩摘要怎么保证上下文不丢失细节？**
核心设计是：旧消息用摘要保留骨架，新消息完整保留细节，两者不矛盾。
    L2 只压缩消息列表的前 40%，后 60% 的最近消息是完整保留的。模型看到的不是"只剩摘要"，而是"摘要回顾早期做了什么 + 最近几步的完整推理链"。摘要提供全局进度感——"我已经搜了三家公司，数据表明今年比去年融资额下降了 30%"——最近消息提供细节——"刚才搜字节返回的具体数字是xxx"。两者互补。
    L3 跨会话记忆那层又补充了一个维度：它检索的是语义相关的历史 exchange，不是全量回放。比如第 13 轮用户问"更新上次研报"，它会召回第 1-10 轮中跟"研报""AI Agent融资"语义相似的 exchange，压缩后注入。模型知道"上次做了什么、结论是什么"，但不用把前 10 轮几万 token 全塞进来。摘要押的是关键信息密度，不是细节完整度——而新消息那 60% 保证的是当下执行精度。

  三层覆盖关系：

  - 最近 60% 消息 → 完整细节，保证当前推理精度
  - 会话内摘要（L2）→ 早期步骤骨架，保证不迷路
  - 跨会话摘要（PGVector）→ 历史经验，保证记忆连续性

  模型同时拥有这三种粒度的上下文，细节丢失风险被三层各自兜住

**简洁版**：并不是只拼接压缩历史 + 最后两条消息。
两套机制分离：
- L1~L3 只压缩本轮内存消息、每轮 think 触发；PG 长期记忆仅新开会话时，语义召回历史生成摘要前置插入。
- 12 轮场景里，前 10 轮仅前 40% 内容压缩成会话摘要，末尾 60% 消息原样留存，11、12 轮新消息完整保存。
- 最终入参：跨会话摘要 + 本轮早期摘要 + 后半段完整原始消息。摘要留存整体结论与脉络，尾部近轮消息保全原始细节，以此兼顾控 token 与上下文完整性。


**既然有 PG 长期记忆，为什么还要做三层上下文压缩？不直接把历史全存向量库，需要时再 RAG 召回，舍弃上下文压缩？**（对标上次 Q9：为什么不做原生记忆替代压缩）
因为这两套系统处理的时间尺度不同，不能互相替代。
L1/L2/L3 负责会话内的“工作记忆”（毫秒级），保证 Agent 在同一个任务中推理连贯、顺序依赖清晰，延迟低、不打断思路。
PGVector 负责跨会话的“长期记忆”（天/周级），用于新会话开始时召回历史信息。
如果砍掉上下文压缩，全走 PGVector，会带来三个问题：
延迟高：每步都要 embedding + 向量检索，累加十几秒，用户卡顿明显。
顺序丢失：语义检索返回的是相似片段，不是上一步的执行状态，推理链会断。
碎片化：连贯的推理过程被打散成独立片段，模型看不到完整因果。
所以两者是分工关系：工作记忆保连贯，长期记忆保持久


**并行 Multi-Agent：JDK21 虚拟线程 + Semaphore 背压，为什么选用虚拟线程不用线程池？**
虚拟线程和 Semaphore 解决的是不同层面的问题，缺一不可。
虚拟线程解决的是“线程不够用”：Worker Agent 是 IO 密集型任务（调大模型、搜索 API 等），大部分时间在等网络响应。传统线程池每个任务占一个 OS 线程，资源重、切换开销大。虚拟线程在阻塞时会自动从载体线程上卸载，极轻量，可以轻松创建成千上万个，非常适合这种 per-task-per-thread 的并发模型。
Semaphore 解决的是“下游扛不住”：虚拟线程虽然能无限创建，但外部 API 有限流、数据库有连接数上限。Semaphore 作为背压阀，控制同时执行的 Worker 数量，防止把下游打爆。
一句话总结：虚拟线程让并发变得廉价，Semaphore 让并发变得可控。两者配合，才能在高 IO 场景下既高效又稳定。

**Semaphore 单 Worker 超时隔离具体实现？**
Semaphore 控制并发：初始容量设5，limiter.acquire() 拿不到许可证的虚拟线程会阻塞排队。因为虚拟线程阻塞不占OS线程，不影响其他逻辑。
超时隔离：每个 Worker 包装成 CompletableFuture，外层调用 get(120, TimeUnit.SECONDS) 等待。超时则 cancel(true) 中断 Worker 执行，并记录失败结果，不影响同一批其他 Worker。
许可证释放：finally 块中释放 Semaphore，并用 acquired 标志位防止未获取许可证时错误释放。
总结：Semaphore 保下游不被冲垮，超时隔离保单任务不拖垮整批，finally 保资源不泄漏。

搜索从 7.4s 降到 3.1s，优化瓶颈在哪？数据怎么统计的？
**数据统计：**我在编排器里内嵌了计时日志。并行执行时，记录每个 Worker 的单次耗时，累加得到“串行推演值”（约8秒）；同时记录整个并行阶段的墙钟（3.1秒）。两者对比就能算出加速比，数字同源可比，不是凭感觉。
**优化瓶颈：**三个搜索任务之间没有数据依赖，但原来的实现是串行的——每次只派一个 Worker，等它跑完再派下一个，中间还夹着 TeamLeader 的 think 循环，总等待时间几乎线性累加。
**优化效果：**改成并行扇出后，墙钟从 7.4 秒降到 3.1 秒，加速比约 2.6 倍。节省的时间就是后面那些任务在串行模式下“干等”的时间，现在被消除了。

发现问题的过程

"我在看 [METRICS] 日志的时候发现一个异常模式：深度研报任务跑 12 步没问题，但一旦任务里涉及网页抓取，L3 硬折叠在第 3-4 步就触发了。L3是最后手段，理论上是逼近窗口上限才用的。这说明消息列表膨胀的速度远超预期——不是渐进积累，而是突然暴涨"

定位根因

"我在 act() 里加了一行临时日志打印每条工具返回的字符数。发现 scrapeWebPage 一次返回 5 万字符的 HTML。两轮抓取就是十几万字符，相当于 2 万+token。我的三层压缩虽然有用，但它的触发线在 3000 token——等你检测到再动手，垃圾已经把预算占满了。就像你家进水了，但水泵要等水淹到膝盖才启动。"
解决思路

"问题本质是防线太靠后。我在 act() 的方法入口加了一层工具输出预截断——每次工具返回结果进入 messageList 后立刻检查，超过阈值直接截断，不等到 compactIfNeeded()的延迟清理。截断阈值按上下文总预算的 20% 来定——一个工具输出不应该吃掉超过 1/5 的上下文，这是从'公平性'出发的：如果一个工具用了 20%，5 个就满了，那后面的推理空间为零。"

如果面试官追问阈值

Q: "5000 这个值怎么定的？有没有做过 A/B 测试？"

思路：
1. 推导过程：6000 token 总预算 ÷ 5（保证至少容得下 5 轮工具调用）= 1200 token/条。中英混合 1 token ≈ 3 chars → ~3600 chars，向上取整 5000 留缓冲。
2. 诚实边界：这个值是理论推导不是实验最优。真正科学的做法是按工具类型差异化——文件读取可能需要更大预算，网页抓取需要更小。但第一版用统一阈值快速验证思路，效果已经很明显（L3触发次数从 2-3 次降到 0）。
3. 埋钩子如果被追问："如果再优化，我会给每个工具在 SKILL.md 里声明自己的 maxResponseChars，工具定义文件本身就声明'我能输出的合理上限是多少'——跟 Skill的声明式理念一以贯之。"
这个回答好在：不回避"没有 A/B 测试"的事实，但展示了完整的推导逻辑和未来的迭代方向。面试官不会觉得你在背标准答案，而是真的在思考"怎么让这个值更合理"。

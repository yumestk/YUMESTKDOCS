**你刚说第一个项目参考了 OpenManus 分层智能体架构，分了 BaseAgent、ReActAgent、ToolCallAgent 这几层，你跟我说说每一层各自负责什么职责？为什么要做分层，不分层直接写会有什么问题？**

BaseAgent 负责状态机、上下文管理和步数控制，不关心具体怎么思考怎么执行。ReActAgent 继承它，把单步执行拆成 think 和 act 两个模板方法，定义了‘先思考再行动’的模式。ToolCallAgent 再继承 ReActAgent，具体实现了 think 里做工具选择、act 里执行工具调用。

分层主要为了扩展性。比如我想加一个只对话不调工具的 Agent，直接继承 ReActAgent 重写 think/act 就行，不用动 ToolCallAgent 的代码。如果不分层，所有逻辑写在一个类里，加新功能时容易互相影响，代码也难以复用和测试。

**你项目里用到了 ReAct 模式，你用自己的话简单讲下，ReAct 核心逻辑是什么？它和普通直接调用大模型接口最大的区别在哪？**

ReAct 的核心逻辑是 Reason + Action，模仿人解决问题的过程：先推理拆解任务，再执行动作（比如调用工具），观察结果后继续下一步，循环直到完成。

它和普通直接调用大模型接口最大的区别有两点：
1. 能获取实时信息：ReAct 可以调用外部工具（搜索、API 等），而普通调用受限于模型训练数据。
2. 推理过程白盒化：ReAct 的每一步思考都可观察、可调试；普通调用是一次性生成黑盒，出问题很难定位。

举个例子，问‘今天北京天气’，普通调用可能说不知道，ReAct 会先推理出需要天气 API，调用后再回答。

**你项目里怎么做 Agent 死循环检测、防止无限循环的？**

我用了两种策略防止死循环：
1. 主动检测：维护一个滑动窗口，记录最近 N 步的 action 名称（比如调用的工具名）。如果连续出现相同的 action 超过阈值（比如 3 次），就判定可能陷入循环。
2. 干预与终止：检测到循环后，先注入一条提示词，告诉模型‘你似乎卡住了，请尝试不同的方法或结束任务’。同时设置全局最大循环步数（如 15 步），无论是否完成都强制终止，返回已有结果。

这两种策略结合，既能智能引导模型跳出循环，又能兜底防止无限消耗 token

**你基于 PGVector 的RAG 完整流程是什么样？多维度过滤又是怎么实现来提升检索精度的？**
RAG 分入库和检索两条链路。

入库：加载本地 Markdown，提取情感标签作为元数据；用大模型生成关键词补强语义；按 Token 切分，控制重叠度，避免切断语义；然后向量化存入 PGVector，用 HNSW 索引和余弦距离做语义匹配。

检索：先做查询改写，提高和知识库的匹配度；然后向量检索；之后做三维过滤——情感标签精准筛选、相似度阈值（比如 0.7）剔除低相关、只取 Top3 片段；最后拼接到提示词，加上拒答兜底，没有命中就礼貌拒答。

我项目里的 RAG 主要分入库和检索两条链路。
首先是文档入库，一共四步：
1. 第一步启动自动加载本地 Markdown 文档，按分隔线做章节拆分，同时从文件名提取单身、恋爱、已婚这类情感标签，存进元数据里。
2. 第二步用大模型给每个文档自动提取核心关键词，补强语义检索容易漏的问题。
3. 第三步自己封装了按 Token 的文本切分器，控制块大小和重叠度，避免把完整语义切断。
4. 最后一步把文本转成向量，存入 PGVector，用 HNSW 索引、余弦距离做语义匹配，检索速度和准确度都更高。
然后是用户提问检索，也是五步：
1. 第一先做查询改写，把用户口语化的问题，改成书面检索句式，适配知识库文案，提升召回。
2. 第二把改写后的问题向量化，去 PGVector 做语义检索。
3. 第三做多维度过滤：先按情感状态精准过滤，缩小检索范围；再设相似度阈值过滤低相关内容；最后只保留 Top3 最相关片段，控制上下文长度、减少噪音。
4. 第四把过滤后的知识片段拼进提示词，给大模型做回答依据。
5. 最后加了拒答兜底，完全没命中相关知识就直接礼貌拒答，避免大模型编造幻觉内容。


**你在智能体项目里提到了记忆持久化，你用了文件系统 + Kryo 序列化，为什么不用 Redis 或者数据库来存对话历史？你当时是怎么考虑的？**

我选择文件 + Kryo 序列化，主要有三个原因：
1. Spring AI 的消息对象结构复杂，用 JSON 序列化会报错，Kryo 能直接完整序列化对象图，兼容性最好。
2. 我想让项目零外部依赖，clone 下来配个 Key 就能跑，不用装 Redis、不用配数据库，降低运行成本。
3. 我基于接口抽象做的记忆模块，未来想换成 Redis 或数据库，只改实现类，不用动业务代码，架构很灵活。

**你项目里用 SSE 做流式输出，为什么不用 WebSocket？SSE 好在哪？**
我项目里用 SSE 而不是 WebSocket，核心是场景匹配 + 成本最低。

我的场景只是服务端单向推送流式结果，客户端不需要主动发消息，SSE 基于 HTTP 长连接，接入简单、兼容性好、不用处理握手和心跳，开箱即用。

而 WebSocket 是全双工协议，需要额外处理连接、重连、代理配置，对我当前场景属于过度设计。

所以我根据实际通信需求，选择了更轻量、更合适的 SSE。

如果未来需要前端主动打断Agent或双向实时通信，我会换WebSocket，但当前场景SSE足够

**一个完整的 SKILL.md 文件长什么样？**
**Agent是怎么"发现"和"加载"一个技能的?** 
设计动机：一开始我的 Agent 配置全是硬编码（工具列表、提示词都写在代码里）。每次想换一套行为，比如从“通用搜索”切到“只做文档分析”，都要改代码重新部署。所以我想把 Agent 的行为定义从代码里抽出来，变成声明式的配置文件。
SKILL.md 文件结构：一个完整的 SKILL.md 分两部分：

上半部分是 YAML frontmatter（--- 包起来），里面定义 id、tools（依赖哪些工具）、platforms（支持什么系统）、systemPrompt（专属提示词）、maxSteps 等。

下半部分是 Markdown，目前给人看的，以后可能让 Agent 自己读。

发现与加载流程：启动时自动扫描 classpath 下的 skills/**/SKILL.md，解析 frontmatter 后注册到 SkillRegistry。当创建 Agent 实例时，会做三层过滤：
1. disabled=true 的跳过
2. platforms 不匹配当前操作系统的跳过
3. 引用的工具不存在的跳过（可选报错）

通过过滤的 skill 会合并工具列表、拼接提示词，然后构造 Agent。

解决的问题：

1. 零代码扩展：新增一个 Agent 行为只需要加一个 SKILL.md 文件，不用改 Java 代码。
2. 环境自适应：同一个 skill 文件在 Mac/Windows 上自动决定是否生效。
3. 工具权限最小化：Agent 只看到当前 skill 需要的工具，降低模型乱选工具的概率，也节省 token

你项目里用到了 PGvector 向量数据库 + 自定义 VectorStore，你给我讲一下：你为什么不直接用框架现成的 VectorStore，要自己自定义实现？整体 RAG      
  检索流程你是怎么做的？                                                                                                                         

⏺ 其实严格来说，我没有自己重新写一个 VectorStore，底层用的还是 Spring AI 现成的 SimpleVectorStore 和预留的
  PgVectorStore。我做的"自定义"主要是在检索管线上加了几个环节，让整套 RAG 不是"把文档塞进去查一下就完事"。

  整体 RAG 流程分五步：

  第一步：文档加载时补元数据。 在 LoveAppDocumentLoader 里加载 document/*.md
  的时候，我从文件名里提取了标签信息（比如文件名的倒数第3第2位字符代表这条知识适合"单身/恋爱/已婚"哪种状态），把它作为 metadata 存进去。另外
  MyKeywordEnricher 会再给每篇文档补一些关键词，让后面检索更容易命中。

  第二步：查询改写。 用户问的问题往往是口语化的，比如"我女朋友老是不回消息怎么办"，直接用这句话去向量检索效果不一定好。我用 QueryRewriter
  调一次大模型，把这句话改写成更结构化、信息密度更高的检索 query，然后再拿改写后的去查。

  第三步：向量检索。 默认用的 SimpleVectorStore，零外部依赖，启动快，适合演示。但我同时写好了 PgVectorStore
  的配置类（只是注解暂时关掉了），等数据量大了切过去只需要改一行注解，维度、距离算法、索引类型这些参数都已经定好了。

  第四步：上下文增强。 检索回来的 topK 文档片段会通过 QuestionAnswerAdvisor 拼到原始 prompt 里，作为 LLM 回答的知识依据。我还预留了一个自定义
  Advisor Factory，可以按用户的情感状态（单身/恋爱/已婚）过滤文档，避免把不相关的知识喂给模型。

  第五步：LLM 生成回答。 模型拿到"原始问题 + 检索到的知识片段 + 系统提示词"一起生成最终回复。

  总结就是：VectorStore 本身是框架现成的，但我围绕它做了文档元数据增强 → 查询改写 → 多后端可切换 → 按场景过滤这一整套检索管线，让 RAG 不是一个
  demo 级的"塞进去查出来"，而是一个可以根据场景调优的流程。

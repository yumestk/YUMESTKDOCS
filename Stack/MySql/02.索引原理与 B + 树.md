MySQL 的索引类型有哪些
- 数据结构角度看:B+树索引; 哈希索引; 全文索引; 空间索引
- 从InnoDB存储方式看:聚簇索引就是主键索引;非聚簇索引,也叫二级索引
- 从索引性质看 : 主键索引,唯一且非空;唯一索引 , 保证列值不重复;普通索引 , 没有唯一约束,只为加速查询; 联合索引 : 多列组合成一个索引,遵循最左前缀原则,列顺序很重要; 全文索引 : 文本搜索用; 空间索引 : GIS数据用

MySQL InnoDB 引擎中的聚簇索引和非聚簇索引有什么区别
- 聚簇索引就是主键索引,叶子节点直接存完整行数据,数据按主键顺序物理存储,一张表只能有一个聚簇索引;非聚簇索引,也叫二级索引,叶子节点之村索引列的值和主键值,查完二级索引还得拿着主键索引去聚簇索引里再查一次,这个过程叫回表
- 核心区别就是存的东西不一样: 聚簇索引存完整数据, 非聚簇索引存主键值. 这导致聚簇索引查询能一步到位, 非聚簇索引可能要多一次回表操作

MySQL 中的回表是什么
- 回表是指用二级索引查询时,因为二级索引只存索引列的值和主键值,拿不到其他字段的数据,如果查询需要其他字段,就得拿着主键值再去聚簇索引查一遍完整的数据行,这个过程叫回表

MySQL 索引的最左前缀匹配原则是什么
- 是指用联合索引时,查询条件必须从索引的最左列匹配,联合索引包含多个列,查询时必须先有第一个列的条件,然后第二个列,以此类推

MySQL 的覆盖索引是什么
- 是指二级索引已经包含了查询所需的所有字段,查询时直接从索引里拿数据,不用再回表去主键索引取完整数据行
- 本质上是一种查询优化,核心就是索引里有的数据就不去表里取了,能省掉回表这一步,I/O少了,速度就自然上去了

MySQL 的索引下推是什么
- 索引下推主要用在联合索引上,把部分查询条件从Server层下推到存储引擎层,在引擎层就把不符合条件的过滤掉,不用再回表查完整数据
- 磁盘/主键索引(完整数据) -> InnoDB引擎层 -> Server层
- 以前是引擎层用索引定位数据,返回主键给Server层,Server层回表拿完整数据,再用剩余条件过滤;有了索引下推,引擎层能直接用索引里的列过滤,不符合条件的不回表,减少了I/O

在 MySQL 中建索引时需要注意哪些事项
- 索引不是越多越好,每个索引都占磁盘空间
- 区分度太低的字段不要建索引,比如性别
- 大字段不要建索引,索引占用空间大,加载到内存耗时
- 写多读少的表要慎重,如果修改频率大于查询频率,索引带来的写入开销可能超过查询收益
- 频繁查询的字段要建索引,where条件里经常出现的字段,多个字段经常一起查可以考虑联合索引,一个联合索引能覆盖多种查询场景
- 排序,分组字段要建索引,order by,group by,distinct后面的字段建索引,能避免临时表和文件排序

MySQL 中使用索引一定有效吗？如何排查索引效果
- 索引不一定有效,建了索引不代表查询就会用,用了索引也不代表查询就快
- MySQL最终是否走索引,靠的是优化器的成本计算
- 排查用EXPLAIN命令:type-访问类型,ALL是全表扫,range范围扫,ref等值匹配,index全索引扫描; key-实际用的索引名称, NULL就是没用上索引; rows-预估扫描的行数,这个数字越大说明查询代价越高

MySQL 中的索引数量是否越多越好？为什么
- 不是越多越好,读的时候能加速查询,写的适合却要付出代价,索引太多,时间和空间成本都会上去
- 从**时间**上看,每次Insert,UPDATE,DELETE都要同步更新索引,删一条name数据,主键索引要改,name字段如果有二级索引也要改,有五个索引就要改五颗B+树,B+树还可能触发页分裂,页合并,写放大问题很严重
- 查询优化器的负担,MySQL执行SQL之前要分析走哪个索引成本最低,索引越多可选项越多,优化器耗时越长,甚至可能因为统计信息不准选错索引
- 从**空间**上看,每个二级索引都是一颗独立的B+树,每个数据页16KB,一个二级索引差不多要占几百MB到几个GB的空间,索引一多,磁盘占用和内存开销都会飙升,Buffer Pool里能缓存的热数据也变少了

请详细描述 MySQL 的 B + 树中查询数据的全过程
- B+树查询数据分两个阶段: 先从根节点往下定位到叶子节点,再在叶子节点内部定位到具体的数据行
- 第一阶段是树的垂直查找,从根节点开始,把查询的键值跟节点中存储的索引键值作比较,用二分法确定落在哪个区间,然后顺着指针往下走到子节点
- 第二阶段是页内查找, 叶子节点是一个16KB的数据页,里面存了很多条记录, InnoDB 用页目录来加速查找, 页目录把记录分成若干个组,每个槽指向一个组的最大记录,先用二分法在页目录里定位到记录所在的组,再沿着组内的单向链表遍历,找到目标记录

为什么 MySQL 选择使用 B + 树作为索引结构
- 核心就一个原因: 磁盘IO次数最少. 数据库的数据存在磁盘上, 磁盘随机读写比内存慢 10 万倍,所以索引结构的设计目标就是尽量减少磁盘访问次数
- B+树能做到这一点靠三个特性
 1. 树矮,B+树是多叉树,一个节点能存几百上千个key
 2. 非叶子节点只存key和指针,不存数据,一个16KB的页能塞进更多索引项,内存里能缓存更多索引,命中率高,磁盘访问少
 3. 叶子节点用双向链表串起来,范围查询时,定位到起点后顺着链表往后扫就行,不用再回根节点重新查找,顺序IO比随机IO快太多

为什么 MySQL 索引用的是 B + 树而不是红黑树
- 核心原因就一个: 磁盘IO次数
- 红黑树是二叉树,每个节点最多两个子节点,存两千万条数据树高大概在24-25层
- B+树是多叉树,每个节点能存上千个索引项,存2000w条数据树高只要3层
- B+树叶子节点用双向链表串起来了,范围查询直接顺着链表扫描,不用反复回溯父节点
- B+树非叶子节点只存索引不存数据,一个节点能塞更多索引项,进一步降低树高

MySQL 三层 B + 树能存多少数据
- 2000w条
- InnoDB默认页大小为16KB,非叶子节点存的是主键和指针,叶子节点存的是完整数据行,假设主键bigint占8字节,指针占6,那么一个非叶子节点能存索引项数量: 16 * 1024 / 14 = 1170
- 假设每条数据记录占1KB,那么一个叶子节点能存的记录数是16/1 = 16条
- 三层B+树结构是: 1个根节点 -> 1170个中间节点 -> 1170 * 1170个叶子节点.   总记录数 = 1170 * 1170 * 16 = 2190w

MySQL 一张表最多可以有多少列
- 4096列
- 实际能建多少主要受两个硬性约束: 行大小限制,存储引擎限制

在什么情况下，不推荐为数据库建立索引
- 数据量小的表,有时候全表扫描比走索引还快,B+树查找,回表这些开销不划算
- 写多读少的表,每次Insert,Update,Delete都要同步维护索引
- 区分度低的字段,如性别
- 查询不走的字段,某个字段从没在where, order by,join条件里出现过,建了索引就是白白占磁盘空间
- 长文本字段,TEXT,BLOB这种大字段,索引Key长度受限,只能建前缀索引,查询效果不好,这种数据有搜索需求应该放ElasticSearch

MySQL 的 Change Buffer 是什么？它有什么作用
- Change Buffer 是InnoDB存储引擎里的一块内存区域,专门用来暂存对二级索引的变更操作,他是Buffer Pool的一部分
- 当你执行Insert,Update,Delete时,如果涉及的二级索引页不在Buffer Pool里,InnoDB不会立刻把索引页从磁盘加载进来,而是先把变更记录到Change Buffer种
- 等后续这个索引页因为查询被读进内存,或者后台线程做Flush时,再把积攒的变更一次性合并进去
- 作用是写入性能大幅提升,原本每次修改都要触发随机磁盘IO把索引页读进来,现在直接写内存就返回了,省掉了大量随机读
- 批量合并更高效, 多次对同一个索引页的修改可以攒在一起,一次性应用,减少重复的磁盘读写次数

MySQL 的 Doublewrite Buffer 是什么？它有什么作用
- Doublewrite Buffer是InnoDB用来解决部分页写入问题的一个安全机制,InnoDB一页是16KB,但操作系统IO单位通常是4KB,这意味着写一个InnoDB页需要4次系统IO
- 如果写一半断电了,比如只写了1个4KB,那这个16KB的页就废了,既不是旧数据也不是新数据,处于半死不活
- Doublewrite Buffer的做法就是: 在把脏页刷到数据文件之前,先把数据写到一个专门的地方做备份,恢复的时候可以从备份区域把完整的页拿出来重新写一遍,保证数据不会损坏

MySQL 的 Log Buffer 是什么？它有什么作用
- Log Buffer是InnoDB用来缓存 redo log 的内存区域,默认大小16MB,它的核心作用是把多次小写入合并成一次大写入,减少磁盘IO次数,提升写入性能
- InnoDB写数据遵循WAL(Write-Ahead Logging)原则,先写日志再写数据页,每次事务提交都要写 redo log 保证崩溃恢复,如果每写一条 redo log 就调一次fsync落盘,系统调用会很大,Log Buffer 就是解决这个问题的,事务执行过程中产生的 redo log 先写到 Log Buffer,等事务提交时再一次性刷盘



> [!NOTE]
> Q :有聚簇索引为何还需要非聚簇索引?
> 1. 聚簇索引一个表只能有一个，存整行数据。
> 2. 业务需要多字段查询，必须建二级索引（非聚簇）。
> 3. 非聚簇索引只存索引列 + 主键，查询非索引字段需要回表。
> 4. 回表确实有性能损耗，所以用覆盖索引、联合索引避免回表。
> 5. 物理存储限制导致不能有多个聚簇索引，所以必须存在非聚簇索引。

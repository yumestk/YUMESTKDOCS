MySQL 中 count (*)、count (1) 和 count (字段名) 有什么区别
- 三者都是统计行数的聚合函数,核心区别在于对 NULL 值的处理方式不同
- count(*) 和 count(1) 会统计所有行,包含NULL值的行; count(字段名)只统计字段不为NULL的行数
- count(*) 和 count(1) 的效率完全一样count(字段名)如果字段没索引就是全表扫描,还要额外判断是否为NULL,所以比前者慢一点,如果字段是主键或者有索引,差距就不大了

**⭐️MySQL 中如何解决深度分页的问题**
- 深度分页的核心问题在于 LIMIT 99999990，10 这种写法会让MySQL扫描前99999990条记录再丢弃，白白浪费大量IO，解决思路就是想办法减少扫描的数据量，常见方案：
1. 子查询优化： 把原本的查询拆成两步，先用子查询在二级索引上快速定位起始ID，再用这个id去主键索引取数据
2. 游标分页： 每次查询都返回当前页的最大id，下次查询时带上这个id作为起点，这种方式利用`id>maxId`直接过滤，MySQL可以从索引定位到起始位置，不用扫描前面的数据，缺点是只能连续翻页，没法跳到10000页
3. 搜索引擎： 把数据同步到ES，用search_after做深度分页，不过ES也有深度分页问题，用search_after配合PIT才好用

⭐️MySQL 中如何进行 SQL 调优
- SQL 调优的核心思路就是减少磁盘IO和避免无效计算， 实际分三步走： 先定位慢SQL，再分析执行计划，最后针对性优化
- 定位慢SQL靠MySQL的慢查询日志， 分析执行计划用EXPLAIN，优化手段主要是这几类：
1. 索引层面优化
- 合理设计联合索引， 利用覆盖索引避免回表，比如查询只需要name和age，那索引就建成(name, age)就不用回表去主键索引拿数据了
- 注意最左匹配原则，where b = 1 吃不到 (a，b，c)这个联合索引
- 避免在索引列上做函数运算，会让索引失效
1. SQL写法优化
- 禁止 select * ，只查必要字段，减少网络传输和内存占用
- 避免 `%like` 前缀模糊查询，必然全表扫描
- 连表查询时检查字符集是否一致，utf8和utf8mb4的字段join会导致隐式转换，索引直接废掉
1. 架构层面优化
- 热点数据直接上Redis缓存，访问频率高但变化少的数据没必要每次都查库
- 大表考虑分库分表，单表超过2000w行查询性能会明显下降
- 读写分离，把查询压力分摊到库
还可以通过业务来优化，比如少展示一些不必要的字段，减少多表查询的情况，将列表查询替换成分页分批查询

**详细描述一条 SQL 语句在 MySQL 中的执行过程**
- MySQL架构分两层: **Server**层负责连接管理, SQL解析, 查询优化这些逻辑处理.**存储引擎层**负责数据的实际存取. 一条SQL从发出到拿到结果,分为这么几步:
1. 客户端先与MySQL建立连接,TCP握手,**连接器**负责验证账号密码,确认你能对哪些库表做什么操作
2. 老版本会先查缓存,看有没有一样的SQL缓存结果,8.0版本已经砍了,因为表一更新缓存就全失效,命中率太低了
3. **分析器**对SQL语句做词法,语法分析. 先把字符串拆成一个个token,识别出SELECT, 表名,列名,where条件元素;再按语法规则检查SQL写的对不对,最后生成一颗抽象语法树
4. **优化器**拿到语法树后,决定用哪个索引,最终形成一个执行计划
5. **执行器**按照计划,调用存储引擎的接口一行行地读数据,做条件过滤,最后把结果返回给客户端
- 连接器建连,分析器解析,优化器决策,执行器调度,存储引擎存取->这就是完整流程

**SQL 中 select、from、join、where、group by、having、order by、limit 的执行顺序是什么**
- 一条`select c1, c2 from table1 as t1 join table2 as t2 on t1.cA = t2.cB where constraint group by column having constraint order by column(asc, desc) limit count`
执行顺序:
- 1. from : 确定数据源,加载表数据
- 2. join : 执行表连接, 合并多表数据
- 3. where : 行级过滤, 筛掉不符合条件的行
- 4. group by : 按指定列分组
- 5. having : 组级过滤, 筛掉不符合条件的组
- 6. select : 选择要返回的列, 计算表达式
- 7. order by : 对结果排序
- 8. limit : 限制返回行数

**MySQL 中的数据排序是怎么实现的**
- 利用**索引天然排序**或者自己动手做`filesort`
- 如果order by的字段正好有索引,且排序方向一致,MySQL直接顺着 B+ 树叶子节点扫描就行, 不用额外排序, 效率很高
- 要是没命中索引,或者索引顺序不对,就得自己`filesort`,这时候MySQL会用`sort_buffer`这个内存缓冲区来排序,数据量小就在内存里搞定,数据量大塞不下就得借助磁盘做外部排序,用归并算法把小块排好序的数据合并成大文件
- 就像整理书架,有索引就直接按标签位置拿,顺序天然就是对的,没索引就自己一本本比大小,书少了内存里排就行,书多了就得搬到仓库分批整理再合并
- 内部排序分为单路和双路排序,单路:把所有的select放到`sort_buffer`排,排完直接返回. 双路排序只放排序字段和行id,排好序后再根据row_id去读其他字段

**MySQL 的查询优化器如何选择执行计划**
- 优化器选择执行计划的核心逻辑是基于成本估算, 选成本低的执行
1. 解析阶段 : 把SQL文本解析成语法树, 检查语法是否正确
2. 预处理阶段 : 验证表和字段是否存在, 检查权限,做一些简单的查询重写
3. 优化阶段 : 生成多个候选执行计划,计算每个计划的成本,选择成本最低的执行
- 成本由两部分 : I/O成本和CPU成本, I/O成本就是从磁盘读数据页的代价,读一页成本记为1;CPU成本是在内存里比较,过滤,排序的代价,处理一行记为0.2;最终成本大致是`读取的页数 * 1 + 扫描的行数 * 0.2`

**从 MySQL 获取数据，是从磁盘读取的吗(buffer pool)**
- 不一定从磁盘读,MySQL的InnoDB引擎维护了一个buffer pool的内存缓冲池, 专门用来缓存数据页和索引页,查询数据时,InnoDB会先看buffer pool里有没有,有就从内存直接返回,没有才去磁盘加载,加载的时候也不是只读那一行,而是把整个数据页读进来,后续访问这个页的其他行也能直接走内存
- 修改数据也一样,直接改buffer pool里的页,改完的页叫"脏页",等后台线程找到合适的时机再刷到磁盘


**MySQL 的存储引擎有哪些？它们之间有什么区别**
- MySQL存储引擎是可拔插的,不同引擎负责数据的存储和读取方式,实际工作中％95以上的场景用的是InnoDB
1. InnoDB: MySQL5.5以后的默认引擎,支持事务,外键,MVCC,适合高并发得到OLTP场景,数据按聚簇索引组织, 主键查询极快
2. MyISAM: 老版本的默认引擎, 不支持事务, 只有表级锁,但读性能好,适合写少读多,对一致性要求不高的场景
3. MEMORY: 数据全放在内存里,速度快但MySQL一重启数据就没了,一般拿来做临时表或者会话级缓存
4. Archive: 专门存归档数据的, 只支持INSERT和SELECT,不支持索引,但压缩率高,历史归档和历史订单这种场景用得上
5. NDB: MySQL Cluster用的引擎, 支持分布式和高可用, 数据自动分片, 适合电信级别的大规模集群


**MySQL 中 InnoDB 存储引擎与 MyISAM 存储引擎的区别是什么**
- InnoDB支持事务和行级锁, MyISAM不支持
1. InnoDB支持ACID事务,实现了四种隔离级别, 通过MVCC来支持高并发,默认隔离级别是可重复读; 支持行级锁,利用行锁加间隙锁在可重复读场景下防止幻读,并发写入性能好; 支持崩溃后的安全恢复,靠redo log的WAL机制;主键索引是聚簇索引,数据和索引放一起,辅助索引只存索引值和主键,查询非索引列时需回表; 支持外键约束,但一般不用,性能差,靠业务代码来保证约束
2. MyISAM基于老的ISAM引擎发展而来,不支持事务(可能有脏数据),行锁,只有表锁,写入时锁整张表,并发写入性能差;非聚簇索引, 索引也是B+树,但数据和索引是分开存储的,主键索引和普通索引结构上没什么区别; 支持全文索引,数据压缩,空间压缩; 不支持崩溃后安全恢复;

为什么在 MySQL 中不推荐使用多表 JOIN
- **核心原因就是数据库往往是系统的瓶颈**, 把复杂的关联操作交给1数据库会占用大量CPU和内存资源,影响其他查询
- 这里的多表一般指超过3个表的join,2个表的join在索引合理,数据量可控的情况下是没问题的
- 数据量小的时候join几个表没什么区别,但当每张表几百万几千万数据的时候,问题就被放大了,join操作需要对每个表进行扫描,匹配,组合,执行计划一旦选错,就可能导致全表扫描,一个慢查询就能把数据库CPU打满,拖垮整个系统

> [!NOTE]
> 可插拔 = 表级别的存储引擎可以自由选择、自由替换，互不影响。不是改全局配置就全库统一，而是每张表自己选引擎。
> 
> 锁粒度:
> 表锁：锁定整张表，并发能力低，实现简单，MyISAM 使用。
> 行锁：仅锁定操作的行，并发能力高，InnoDB 使用，支持事务。
> 
> MVCC = Multi-Version Concurrency Control
> MVCC:多版本并发控制，通过保存数据的历史版本，实现读写不阻塞，提高数据库并发能力，是 InnoDB 实现事务隔离级别的核心机制。
>
> WAL 机制：先写日志，再写磁盘。
> redo log 保证事务提交后，即使数据库崩溃也能恢复数据，实现持久性。
>
> 慢查询（Slow Query）
> 执行时间超过指定阈值的 SQL 语句，就叫慢查询。
>
> filesort:
> filesort = MySQL 不得不自己在内存 / 磁盘上做排序，而不是直接用索引排好的顺序取数据

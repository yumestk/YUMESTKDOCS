Redis 支持事务吗？如何实现？
- Redis 支持事务,但它和MySQL事务是两码事. Redis事务只保证命令打包执行不被其他客户端插队,不支持回滚
- Redis事务用MULTI, EXEC, WATCH, DISCARD四个命令实现
1. MULTI 开启事务,之后的命令不会立即执行,而是进入一个队列排队
2. 队列里可以塞任意多条命令, Redis会回复QUEUED标识入队成功
3. EXEC一次性把队列里的命令全部执行,期间不会被其他客户端的命令打断
4. 如果中途不想执行了,用DISCARD丢弃整个队列
5. WATCH可以在MULTI之前监视某些key,如果这些key在EXEC之前被别人改了,整个事务作废
- 最重要的一点是: MySQL事务执行一半出错可以回滚,Redis不行. Redis事务里某条命令执行失败,剩下的命令照样继续跑,已经执行的不会撤销

Redis 事务与关系型数据库事务的主要区别是什么？
- 关系型数据库讲究 ACID, 而Redis事务只能算是批量执行命令的机制,连回滚都没有
- 关系型数据库的ACID, 原子性: 事务的操作要么全成功,要么全失败回滚,不存在中间状态; 一致性: 事务执行完数据库里的数据还是保持一致的,不会出现数据错乱; 隔离性: 多个事务并发执行互不干扰, 还能设置不同隔离级别; 持久性: 事务一旦提交, 数据就永久保存了, 断电也不丢
- Redis事务对ACID的支持情况: 原子性: 只保证命令连续执行,中间不被打断. 但如果某条命令执行失败,后面的命令照样走,没有回滚; 一致性: 没有回滚机制就没法保证一致性, 中间出错就乱了; 隔离性: Redis是单线程执行命令的,事务执行期间别的命令插不进来,天然串行化, 但没有隔离级别选; 持久性: 取决于RDB和AOF的配置,理论上还是可能丢数据

Redis 的 Lua 脚本功能是什么？如何使用？
- Redis允许用户在Redis服务器端执行自定义的Lua脚本,用来实现原子操作和复杂逻辑
- 核心点包括: 
1. 原子性: Lua脚本的所有命令在执行过程中是原子的,避免了并发修改带来的问题
2. 减少网络往返次数: 通过在服务器端执行脚本,减少了客户端和服务器之间的网络往返次数, 提高了性能
3. 复杂操作: 可以在Lua脚本中执行复杂的逻辑,比如批量更新,条件更新等,超过了单个Redis命令的能力
- 最典型的场景就是分布式锁, 释放锁的时候要判断锁是不是自己的,是的话再删除, 如果用两条命令分开执行,中间可能被别的请求插进来,导致误删别人的锁,用lua脚本把判断和删除包在一起,就不会出这种问题
- Lua脚本本身不具备原子性,但Redis是单线程执行命令的,它把Lua脚本当成一个命令跑,执行期间不处理任何其他请求, 也就使得执行过程中不会被其他命令打断,看起来好像有了原子性

Redis 的 Pipeline 功能是什么？
- Pipeline让客户端把多条命令打包一次性发给Redis,不用等每条命令的响应就能继续发下一条,最后再一起收结果,核心价值就是减少网络往返次数,性能提升明显
- 正常执行100条命令,客户端要和Redis来回通信100次,用pipeline的话,100条命令一次性发过去,Redis按顺序执行完,结果一次性返回,网络往返只有一次,假设网络延迟为1ms,100条命令正常执行光网络等待就要100ms,用pipeline只要1ms
- Pipeline的好处:节省网络往返时间,RTT从N次降到1次; 减少系统调用的上下文切换开销,100条命令只需要1次用户态到内核态的切换

Redis 中原生批处理命令（MSET、MGET）与 Pipeline 的区别是什么？
- MSET,MGET是Redis内置的原子性批处理命令, 一条命令搞定多个key的读写; Pipeline是客户端层面的批量发送机制,把一大堆命令打包发出去,让Redis挨个执行再一次性返回结果
- 两者区别在于原子性和适用范围: MSET和MGET在Redis服务端是单条命令执行, 天然原子性,中间不会插入其他命令,但只能做批量读或写,想混用INCR,LPUSH就没戏了; Pipiline就灵活多了,什么命令都能往里塞,但本质上还是多条独立命令,只是帮你省了网络往返,期间可能被插队; 网络开销上两者都是一次往返,但MSET/MGET在服务端只有一次命令解析开销,Pipeline有N次,数据量不大可以忽略,几万条命令Pipeline的解析成本就能感知到了

Redis 的订阅发布功能是什么？你了解吗？
- Redis的Pub/Sub是一条消息通信机制,用于在不同客户端之间实现消息的实时传播和广播,客户端可以订阅一个或多个频道,当有其他客户端向这些频道发布消息时,所有订阅了该频道的客户端都会立即收到消息
- 工作流程: 发布者通过PUBLISH命令往某个频道发消息; Redis找到所有订阅了这个频道的客户端; 把消息转发给这些客户端
- Redis不会存储消息,它只是个转发通道,消息发出去之后,如果订阅者不在线,这条消息就永远丢了
- 基本命令就四个: SUBSCRIBE channel: 订阅某个频道; PUBLISH channel message: 往频道发消息; UNSUBSCRIBE channel: 取消订阅; PSUBSCRIBE pattern模式匹配订阅

Redis 中的 Geo 数据结构是什么？
- Redis中的Geo是专门用来存储和查询地理位置信息的数据结构,从Redis3.2版本开始支持, 你可以往里面存经纬度坐标,然后做附近的人,"周围3公里有哪些门店"这类查询
- 底层实现其实是Sorted Set,每个位置点用Geohash算法编码成一个分数,利用有序集合的范围查询能力来实现地理搜索

如何使用 Redis 快速实现排行榜？
- 用Redis的Sorted Set来实现排行榜是最合适的选择. Sorted Set里每个成员都绑定一个score分数,Redis会按score自动排序,而且底层用跳表实现,插入,删除,查排名都是O(logN), 百万级数据也能扛住

如何使用 Redis 快速实现布隆过滤器？ (Bloom Filter)
- 布隆过滤器是一种概率型数据结构,用来判断某个元素是不是在集合里, 它的特点是: 说不在就一定不在,说在则可能误判. 用Redis实现布隆过滤器主要有两种方式: 位图手动实现或者用官方的RedisBloom模块
- 位图方式就是用Redis的SETBIT/GETBIT命令,自己管理**哈希函数和位数组**. 添加元素时, 用k个哈希函数算出k个位置, 把这些位置都设成1; 查询时同样算k个位置, 只要有一个是0就说明元素肯定不存在,全是1就说明可能存在
- 误判发生在不同元素的哈希值产生冲突时

如何使用 Redis 统计大量用户唯一访问量（UV）?
- Redis中的HyperLogLog就是干这个的,用极少的内存统计海量数据的去重数量. 它是一种基数估算的概率型数据结构,每个key固定只占12kb内存,哪怕你往里塞1亿个用户ID,内存占用也不会涨,代价是有0.81%左右的误差,不过对于UV统计这种场景,差个几百几千也能接受
- 用Set也能做去重统计, 但1000w用户ID光存Set就得几百MB,hyperLogLog只要12kb,差了四个数量级
- HyperLogLog和Set对比: 
1. Set精确统计,内存随数据量线性增长,1000w用户约400MB
2. HyperLogLog: 近似统计, 固定12KB内存,误差0.81%

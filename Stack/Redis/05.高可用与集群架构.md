Redis 主从复制的实现原理是什么？
- Redis的主从复制简单来说就是让从节点(Slave)认一个主节点(Master)做大哥,大哥有什么数据,小弟就跟着同步一份,这样既能读写分离减轻大哥压力,也能在数据丢了时有个备份
- 三个阶段:
1. 建立连接与全量同步: 当从节点第一次连上主节点,会发送PSYNC命令,因为是第一次,主节点会执行一次全量复制,具体就是主节点会在后台生成一份RDB快照文件发给从节点,从节点拿到后先清空自己的旧数据,然后加载这份快照
2. 命令传播: 全量同步完成后,主从之间会建立长连接,以后主节点每收到一个写命令,就会异步地发送给从节点,从节点跟着执行就好了,这期间他们还会互相发心跳包(Ping/Ack)来确认对方还活着
3. 断线重连与增量同步: 网络总是不稳定的,如果从节点掉线一小会又连上了,重新搞一次全量同步太浪费资源了,所以Redis2.8以后引入了增量同步,主节点内部有一个环形的缓冲区叫 `repl_backlog_buffer` 积压缓冲区, 如果从节点连上来,告诉主节点: 我刚才读到偏移量 1000 了, 主节点一看挤压缓冲区里1000之后的数据还在,会只把短线期间这部分数据补发给从节点,这就叫增量复制,断线太久,缓冲区被覆盖了,那就只能全量同步

Redis 主从复制的常见拓扑结构有哪些？
- 三种: 一主多从, 树状级联和主主结构
- 一主多从: 最常见的结构,一个主节点挂多个从节点,写操作全走主节点,读操作分散到各个从节点,实现读写分离,主节点挂了的话,从节点可以顶上来,适用于缓存等读多写少业务
- 树状级联: 从节点当其他从节点的主,形成一个层层往下传递的结构1,好处是主节点只需要同步给少数几个一级从节点,一级从节点再往下同步,主节点的复制压力小很多,缺点是链路越长,数据延迟越大,最底层从节点拿到最新数据可能慢个几百毫秒,适用于从节点超过10个,避免主节点网卡CPU过载
- 主主结构: 两个主节点互相同步,写操作可以打到任意一个主节点上, 但Redis原生不支持主主复制,搞得自己处理数据冲突,两边同时写一个key就炸了,实际生产环境很少用,除非有特殊的多活场景配合业务层面的冲突规避

Redis 复制延迟的常见原因有哪些？
- Redis 主从复制会有延迟从节点的数据比主节点慢一拍,在读写分离场景下,刚写进去的数据去从节点查可能查不到,这就是复制延迟带来的问题
- 常见原因:
1. 网络问题: 带宽不足(监控流量峰值), 网络抖动/丢包(网络稳定性), 跨机房部署延迟高
2. 主节点瓶颈: 写入QPS过高(优化写入操作), RDB/AOF持久化占用资源(磁盘IO争用),大Key写入阻塞(避免大Key,拆分)
3. 从节点瓶颈: 机器配置低(升级硬件), 磁盘IO跟不上(检查磁盘性能,使用SSD), 同时挂载过多从节点(限制每个主节点的从节点数量)
4. 复制机制问题: 缓冲区溢出触发全量复制(增加缓冲区大小), 复制积压缓存区配置过小(调整1配置)

Redis 的哨兵机制是什么？
- Redis Sentinel 就是一套 高可用监控系统, 专门盯着主从集群, 主节点挂了能激动把从节点提上来顶替,客户端也能自动感知到新主节点的地址
- 哨兵干三件事: 
1. 监控: 每秒给所有Redis节点发PING,看谁没响应
2. 故障转移: 主节点挂了,从从节点选一个升级成1新主节点
3. 通知: 把新主节点的地址推送给客户端和其他从节点
- 典型的哨兵架构:3个哨兵节点组成集群,互相通信; 1个Redis主节点处理写请求; 2个Redis从节点做读请求和数据备份; 哨兵节点监控所有Redis节点的状态; 主节点挂了,哨兵选出新节点并通知客户端
- 生产时至少部署3个哨兵节点,单数个,这样投票的时候不容易出现平票

Redis 集群的实现原理是什么？
- Redis集群主要解决了单机内存和并发的瓶颈,它通过多个Redis实例组成,每个实例存储不同的数据分片
- 实现原理用三个关键词来概括:分片,Gossip(闲聊)协议,去中心化
- 数据怎么存: Redis集群引入了哈希槽的概念,把整个数据集群划分为16384个槽,集群里每个主节点负责维护一部分槽, 存一个key时,Redis先对Key算一个CRC16值,然后对16384取模,算出它属于哪个槽,再把数据存到负责这个槽的节点上,这样做的好处就是扩缩容方便,要加新节点只需把其他节点身上的一部分槽过户给新节点就行了,不需要像传统哈希那样重新计算所有数据的位置
- 节点之间怎么联系: 集群的节点是去中心化的,没有所谓的大脑,它们通过Gossip协议互相通信,每个节点定期选择几个邻居,交换彼此的状态信息,比如 "我负责哪些槽", "我是否还活着",这样只需很短的时间,集群的所有节点就能达成一致,知道整个集群的拓扑结构
- 客户端怎么找数据: 客户端可以连接集群的任意一个节点,如果客户端要找的Key刚好在这个节点上,那就直接返回; 如果不在,这个节点不会充当代理去转发请求,而是会返回一个MOVED错误,告诉客户端: 这个Key不归我管,你去连IP:Port去

在 Redis 集群中，如何根据键定位到对应的节点？
- Redis Cluster 把数据分布到 16384 个哈希槽里, 每个Key通过CRC16算法算出哈希值,再对16384取模,得到槽位编号,每个节点负责一部分槽位,知道key落在哪个槽,就知道数据在哪个节点上
- 公式: slot = CRC16(key) % 16384

Redis 集群会出现脑裂问题吗？
- 会出现, 而且这是分布式系统的经典问题,当发生网络分区时,Redis集群可能出现多个主节点同时提供写入服务,导致数据不一致
- 举个典型场景: 假设一个主节点和它的从节点之间网络断了,哨兵检测到主节点失联后会选举一个从节点升级为新主,但原来的主机点其实还活着,只是网络不通而已,这时候就出现了两个主节点,客户端往哪个主写数据都会导致数据不一致; 更麻烦的是,等网络恢复后,旧主节点会被降级为从节点,它会从新主节点同步数据,之前客户端写到旧主的数据就直接丢了
- Redis提供了两个参数来缓解脑裂问题: min-slaves-to-write: 主节点必须有至少N个从节点确认才能执行写操作; min-slaves-max-lag: 从节点最大延迟秒数,超过这个值就不算有效从节点

Redis Cluster 模式与 Sentinel 模式的区别是什么？
- Sentinel只管故障转移,Cluster既管故障转移又管数据分片
- Sentinel是哨兵模式,专门盯着主从复制架构里的主节点,主节点挂了,哨兵会把某个从节点提升为新主节点,保证服务不中断,但数据还是全量存在每个节点上,没法水平扩展
- Cluster是官方的集群方案,把数据自动打散到多个节点上存储,每个节点只存一部分数据,同时Cluster内置了类似哨兵的故障检测和自动故障转移机制,某个主节点挂了,它的从节点会自动顶上,不需要额外部署哨兵
- 选型: 数据量不大,单机能抗住,只是想要高可用,用主从 + Sentinel, 架构简单,运维成本低; 数据量大,单机内存不够用,或者写入压力大需要分散,用Cluster,能水平扩展,加节点就能扩容

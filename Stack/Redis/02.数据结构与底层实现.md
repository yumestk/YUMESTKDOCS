**Redis String 类型的底层实现是什么？**
- 用的是SDS(Simple Dynamic String),简单动态字符串
- SDS的设计核心是在字符数组前面加了个头部结构,记录当前长度len和分配的空间alloc
- SDS结构包含三个核心字段:len记录当前字符串长度,alloc记录分配的总空间,buf存实际数据
- Redis根据存储内容做了编码优化: 能解析成整数的,直接用int编码,把数字存在指针位置,连SDS都不用分配; 44字节以内的短字符串用embstr编码,redisObject和SDS分配在一块连续内存,一次malloc搞定;超过44字节的长字符串用raw编码,redisObject和SDS分开存,方便扩容

**Redis 中跳表的实现原理是什么？**
- 跳表本质上是多层链表,底层链表保存所有元素,上层链表是下层链表的子集,通过这种分层索引结构把链表的O(n)查找优化到O(logn)
- 查找的时候从最高层开始,先往右走,遇到比目标值大的就往下走一层,重复这个过程直到找到或确定不存在
- 插入的时候先用查找的方式定位到插入位置,然后随机决定要建几层索引,Redis用25%的概率往上加一层,所以大部分节点都在底层
- 删除就是先找到节点,然后把这个节点在各层的前后指针都接上,跟普通链表删除一样,只是要在多层都操作一遍

**Redis 的 hash 是什么？**
- Redis的hash是一种键值对集合,能把多个字段和值存在同一个key下面,特别适合存对象的属性. 比如存用户信息,用 `user : 1001` 作为key,里面放name, age, city这些字段,改单个字段不用整体覆盖,比用String存Json灵活
- Hash底层有两种编码方式,数据量小时用listpack存储,紧凑省内存;数据量大了自动切换成hashtable,查询效率O(1), 切换阈值默认是512个字段或者单个值超过64字节
- 常用操作: HSET写字段, HGET读字段, HMSET批量写,HMGET批量读,HDEL删字段,HINCRBY对数值字段做原子加减

**Redis Zset 的实现原理是什么？**
- ZSET的底层是跳表 + 哈希表双结构配合,跳表按score排序,支持O(logN)的插入,删除和范围查询; 哈希表存member到score的映射,支持O(1)的单点查分

**为什么 Redis Zset 用跳表实现而不是红黑树？B + 树？**
- 实现简单,范围查询高效,结构灵活可调
- 不用红黑树是因为跳表实现简单,跳表就是多层链表,通过概率算法动态生成索引层级,没有左旋右旋,代码写起来清爽,红黑树要维持平衡,插入删除都要旋转,代码量和理解成本都高不少; 并且范围查询高效,跳表定位到起点后,直接沿着底层链表往后遍历就行,缓存局部性很好,红黑树从结构上就不支持这种操作,要做范围查询就得中序遍历; 结构灵活, 跳表的层数是动态的,节点要升几层完全靠概率决定,数据量大就多几层,数据量小就少几层,自动适应,红黑树结构是定死的
- B+树是为磁盘设计的,放内存里反而不划算: 实现复杂度高, B+树插入和删除要处理节点分裂,合并,重新平衡,代码复杂度远高于跳表,跳表的插入删除本就是链表操作+随机层数; B+树真正优势在于减少磁盘IO, B+树高扇出是为了让树更矮,减少磁盘寻道次数,非叶子节点只存索引能让内存放下更多索引信息,适合MySQL这种海量数据存磁盘的场景,Redis数据全在内存,用不上这个优势

**Redis 中的 Ziplist 和 Quicklist 数据结构的特点是什么？**
- Ziplist是一块连续内存,把所有元素紧凑地挨在一起存,省内存但不适合大量数据
- Quicklist是Redis3.2之后List的默认实现,本质是双向链表穿起来一堆小的Ziplist兼顾了内存效率和操作性能

**Redis 的 ListPack 数据结构是什么？**
- ListPack是Redis 6.0 引入的一种紧凑型序列化数据结构, 专门用来替代ziplist, 它把数据直接按字节序列排序存储,不走Redis常规的对象模型,内存占用极低; Redis 7.0 之后, List, Hash, ZSet在数据量小的时候底层都改用ListPack了,把ziplist换了
- ListPack的结构很简单: header占6字节,前4字节存总长度,后2字节存元素个数; 中间是一个个element紧凑着排列; 末尾一个0xFF字节标识结束

**Redis 中 EMBSTR 对象的阈值设置为何为 44? 其调整历史是什么？**
- 44字节这个阈值是被内存分配器的分配单元给定死的
- Redis用的是jemalloc, 最小分配单元是64字节,超过64字节就得分配更大的块,所以Redis要把EMBSTR编码的字符串塞进64字节里
- 64字节怎么分配: redisObject结构体占16字节, type占4位,encoding占4位,lru占24位,refcount占32位,ptr指针占64位,加起来128位就是16字节
- sdshdr8结构体占3字节: len占1字节记录已使用长度,alloc占1字节记录分配长度, flags占1字节标记类型
- 字符串末尾的 '\0' 占1字节; 剩下就是64-16-3-1 = 44字节,刚好能存44个字符

**Redis List 类型的常见操作命令有哪些？**
- Redis List 是一个双端链表结构的字符串列表, 支持从两头插入和弹出元素
- 写入类命令: LPUSH从左边塞元素, RPUSH从右边塞元素, list不存在会自动创建; LPOP从左弹出, RPOP右弹, 弹出来的元素会从list里删掉; BLPOP 和 BRPOP 是阻塞版本,list为空的时候会等着,直到有数据或者超时
- 读取类: LRANGE 按范围取元素; LINDEX按下标取单个元素,下标从0开始; LLEN拿list长度
- 修改类: LSET按下标改元素值; LREM删除指定数量的匹配元素; LTRIM只保留指定范围的元素, 其他全删掉

**如何在 Redis 中实现队列和栈数据结构？**
- 用Redis的List类型就能直接实现队列和栈, 核心就是控制从那边进,从哪边出
- List的LPUSH/RPUSH/LPOP/RPOP时间复杂度都是O(1), 底层是quicklist结构,插入删除非常快

**Redis 源码中有哪些巧妙的设计，举几个典型的例子**
- Redis能做到单机10万+QPS,靠的就是源码里一堆精心打磨的细节, 最典型设计有这几个: SDS字符串, 压缩列表ziplist, 渐进式rehash,共享对象池,惰性删除+定期删除
- SDS字符串: C语言原生字符串有几个硬伤, 获取长度要O(n)遍历,拼接容易溢出,不能存二进制数据, Redis搞了个SDS结构, 直接把长度和剩余空间记在头部
```
struct sdshdr {
    int len;    // 已用长度
    int free;   // 剩余空间
    chat buf[]; // 实际数量
}
好处很直接:
1. 取长度O(1), 不用遍历
2. 追加字符串时先检查free够不够,不够就扩,杜绝溢出
3. 不依赖\0判断结尾,二进制数据随便存
4. 字符串缩短后空间不立即释放,留着下次用,减少内存分配次数
```
- 压缩列表 ziplist (Redis 7.0 之后 ziplist 已经被listpack替代) : 当List和Hash里元素少,每个元素又短的时候,Redis不用链表,用一整块连续内存把数据挤在一起,每个节点前面记着前一个节点的长度,往前往后遍历都很快,还省掉了指针开销
- 渐进式rehash: 哈希表扩容是个大活, 一次性搬几百万个key能把主线程卡死, Redis做法是准备两张表ht[0]和ht[1],扩容时新数据往ht[1]写,同时每次增删改查顺手搬几个旧节点过去,等ht[0]清空了,指针一换就完事
- 共享对象池: 0到9999这一万个整数在业务里出现频率极高,Redis启动时就把它们全部预创建好放在池子里,后续用到直接拿引用,不用反复malloc/free, 一个计数器加1减1,底层压根不涉及内存分配
- 过期键处理: Redis没有专门的线程去扫描过期键,一是惰性删除: 访问某个key时顺手检查了,过期了就删. 二是定期删除: 每100ms随机抽20个设置了过期时间的key,删除其中过期的,如果过期比例超过25%就再来一轮, 这样既不会漏删,也不会因为全表扫描拖慢响应

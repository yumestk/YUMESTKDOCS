**Redis 通常应用于哪些场景？**
- 主要作用就是作为缓存提升性能和用于一些分布式协调场景
- 第一, 最核心的就是做缓存,因为MySQL是存在磁盘的,并发一高就容易挂.Redis是纯内存操作,读写速度非常快. 我们通常使用String类型缓存用户信息,Session会话或者缓存一些复杂的查询结果,比如热点商品的详情页,这样能挡住绝大部分请求,保护后端的数据库
- 第二, 解决并发问题的分布式锁, 现在基本上都是微服务集群部署,java自带的synchronized锁不住别的机器上的线程,我们利用redis的setnx命令或者redisson框架.**因为redis本身是单线程处理命令的, 天然能保证原子性**,谁先写入谁就先拿到锁,非常适合用来防止超卖和重复提交
- 第三, 高频的计数器和限流, 如果用MySQL做计数,比如文章阅读量,点赞数,每次都要update数据库,锁竞争严重. 而redis基于内存性能高很多, 并且redis是单线程原子性的,用icnr命令自增, 还不会算错. 另外还可以用lua脚本配合redis做api限流, 比如一分钟只能调100次
- 第四, 复杂的排行榜业务, 如果在数据库里做实时排名, 需要全表扫描并排序, 数据量一大数据库就崩了. 而Redis有个专门的数据结构叫ZSet, 它写入时就自动排好序了,做直播间贡献榜,战力榜的时候用ZADD和ZRANGE就能毫秒级拿到前十名.
- 第五, 轻量级消息队列, 有时候不想引入Kafka那么重的消息中间件,只是做一个简单的异步处理. 用List结构的LPUSH和RPOP来做简单的任务队列,当然这是消息可靠性要求不高的情况.

**Redis 为什么这么快？**
- 纯内存操作,单线程 + IO多路复用,高效的数据结构
- 1. Redis把数据全放在内存里,基本不和磁盘打交道,除了持久化的时候
- 2. Redis用单线程执行命令,没有锁竞争,没有上下文切换,网络IO用epoll多路复用,一个线程能盯着几万个socket连接,哪个连接有数据就处理哪个,不用傻等
- 3. Redis的数据结构都是为了快而生的;String底层是SDS,O(1)获取长度;Hash小数据量用ziplist省内存,大数据量用hashtable保证O(1)查询;ZSET用跳表,插入查询都是O(logN)

**为什么 Redis 设计为单线程？6.0 版本为何引入多线程？**
- Redis用单线程是因为Redis的瓶颈不在CPU,而在内存和网络IO, 大多数业务单线程就够用了
- 6.0 引入多线程,主线程只负责执行命令,是因为业务量上来之后,**网络IO成了瓶颈**,数据从内核拷贝到用户空间这一步是同步的,并发量高的时候,单线程处理不过来,多线程分摊网络读写的工作,命令执行还是单线程,既提升了吞吐量,又不用担心线程安全问题

**Redis 中常见的数据类型有哪些？**
- 主要五种:String, List, Hash, Set, ZSet
- String 是最基本的类型,能存文本,数字,二进制,最大512MB.典型场景就是缓存用户会话,页面数据或者做计数器,比如文章阅读量,点赞数,直接INCR命令原子递增就行
- Hash本质上是个键值对集合,适合存对象属性.比如存商品详情,商品ID作为Key,价格,库存,名称这些字段都塞进一个Hash里,改单个字段不用整体覆盖
- List是有序的字符串列表,底层是双向链表,支持双端操作,最常用的场景就是消息队列,LPUSH生产消息,RPOP消费,简单的生产消费者模式就搞定了
- Set是无序且元素不重复的集合,查找和去重效率高.适合标签系统,记录某个页面的独立访客,共同关注等需要去重或集合运算的场景
- ZSet和Set类似,但每个元素都带一个score分数用来排序,底层用跳表实现.最典型就是排行榜,按score排序后直接取TopN
- BitMap用位来存数据,空间利用率高; HyperLogLog是个概率型数据结构,专门用来估算基数,适合网站UV这种对精度要求不高但数据量大的场景
- GEO用来存地理位置信息,支持经纬度存储和空间查询,底层是ZSet实现.典型场景就是附近的人,外卖配送距离计算

**Redis 和 Memcached 有哪些区别？**
- 这俩都是内存级别的缓存系统,但Redis功能更加丰富,Memcached更轻量纯粹
- 数据结构方面Redis如上,但Memcached只支持简单的Key-Value字符串存储
- 持久化方面,Redis提供RDB快照和AOF日志两种方式,服务重启数据还在;Memcached压根没有持久化,一重启数据全没了
- 分布式方面,Redis原生支持主从复制,哨兵和Redis Cluster集群模式;Memcached没有分布式逻辑
- 功能方面Redis支持发布订阅,Lua脚本,事务,Stream消息队列;Memcached就是纯粹的缓存,啥扩展功能都没有

**Redis 字符串类型的最大值大小是多少？**
- 512MB,这是Redis源码里写死的硬限制
- 无论是网络传输,内存分配还是字符串操作,大字符串都会增加Redis服务器的负载,且过大的字符串在GET,SET,APPEND,STRLEN等操作都会导致性能瓶颈,主从同步也有延迟风险,所以官方给字符串大小做了限制,防止单个键值对占用过多的内存,影响整体稳定性和性能
- 实际生产中,单个字符串**10KB**以内是比较健康的,超过100KB就要考虑拆分了,512只是保护,不是让你真用这么大

**你在项目中使用的 Redis 客户端是什么？**
- Jedis是老牌客户端,API直接对应Redis命令,上手快,但它是同步阻塞的,线程也不安全,每个线程得用自己的连接,需要线程池
- Lettuce基于Netty实现,天生支持异步和响应式编程,一个连接可以被多个线程共享,高并发下更省资源
- Redisson走高级封装路线,直接把分布式锁,分布式集合,限流器这些常用组件都包装好了,开箱即用
- 简单业务用Lettuce就够了,SpringBoot默认集成不用额外配;需要分布式锁,限流这些能力直接上Redisson;老项目或者团队熟悉Jedis继续用也没问题,配好连接池一样稳定 

**Redis 性能瓶颈时如何处理？**
- Redis性能瓶颈的处理核心就是分摊压力,从单机到集群逐级递进
- 垂直扩容: 先看单机资源有没有用满,内存不够就加内存,CPU频率能提就提;一台64GB内存的机器跑Redis,能撑住大部分中小业务
- 读写分离: 写请求走主节点,读请求分流到从节点;一主三从的架构,读QPS直接翻3倍,适合读多写少的场景,比如商品详情页
- Redis Cluster分片: 数据量超过单机内存上限,或者写请求压力太大就得分片集群
- 多级缓存: 在应用层加一层本地缓存,Caffeine或者Guava Cache都行,把热点数据兜在本地,压根不走网络,像秒杀场景,库存这种热Key必须本地缓存顶着

> epoll 多路复用是什么？
> 一个线程，同时监听很多个网络连接（socket），哪个连接有数据来了就处理哪个，没数据就不浪费时间。
>
> socket 连接 = 网络上两台机器之间的一条双向通信通道。
>
> Redis 的每种数据结构都有讲究:
> 
> SDS (Simple Dynamic String) : 比 C 语言原生字符串多存了一个 len 字段, 获取长度 O(1), 还支持二进制安全, 能存图片、序列化对象。
> 
> ziplist: 紧凑的连续内存结构, 省内存但查询是 O(N)。Hash、List、ZSet 在数据量小的时候都用它。
> 
> quicklist: Redis 3.2 之后 List 的底层实现, 把多个 ziplist 用双向链表串起来, 兼顾内存和性能。
> 
> skiplist: ZSet 的底层之一, 查询、插入、删除都是 O(logN), 实现比红黑树简单, 范围查询更方便。
>
> CPU与多线程
> CPU 是资源，线程是使用资源的单位,CPU 提供计算能力，多线程负责把任务分配到多个核心，让 CPU 能力被充分利用。瓶颈在 CPU 时，多线程才能提速
>
> Redis中字符串类型的底层实现用的是SDS(Simple Dynamic String)是Redis专门为字符串存储优化的结构
>
> fork就是分叉,复制一份然后各玩各的,不影响
>
> QPS = Queries Per Second = 每秒能处理的请求数就是每秒扛多少个查询，是衡量 Redis、数据库、接口性能最核心的指标。QPS越高,并发性能越好

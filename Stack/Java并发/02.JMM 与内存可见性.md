什么是 Java 内存模型（JMM）？
- Java内存模型是JVM定义的一套规范，规定了多线程程序中变量如何在内存中存储和传递，约定了线程何时从主内存读取数据，何时把数据写回内存
- JMM的核心目标是确保多线程环境下的可见性，有序性和原子性，屏蔽掉硬件和编译器优化带来的不一致问题
1. 可见性： 一个线程对变量的修改能及时被其他线程看到，volatile关键字就是用来保证可见性的，强制线程每次读写都直接跟主内存交互
2. 有序性： 线程执行操作的顺序，JMM允许指令重排序来提高性能，但通过happens-before关系保证跨线程的有序性
3. 原子性： 操作不可分割，执行过程中不会被打断，synchronized关键字能保证代码块的原子性
- JMM的抽象内存模型：
1. 主内存存放共享变量，所有线程都能访问
2. 每个线程有自己的本地内存，存放共享变量的副本
3. 线程对变量的操作必须在本地内存中进行，不能直接操作主内存
4. 线程间变量传递必须通过主内存完成

什么是 Java 中的原子性、可见性和有序性？
- 原子性，可见性和有序性是Java并发编程的三大核心特性，任何并发bug基本都能归到这三类
- 原子性指一个操作要么执行完，要么没执行，中间不会被其他线程打断，比如i++这个操作看着像一行代码，实际上是读取，加1，写回三个步骤，多线程环境下就可能出问题
- 可见性指一个线程修改了共享变量，其他线程能立刻看到最新值，CPU有自己的高速缓存，线程修改的值可能还躺在缓存里没刷回主内存，别的线程就读到了旧值
- 有序性指程序执行顺序和代码写的顺序一致，编译器和CPU为了性能会对指令做重排序，单线程下没问题，多线程可能会出诡异的bug

什么是 Java 的 happens-before 规则？
- happens-before是JMM定义的一套规则，用来约束多线程环境下操作的可见性和顺序性，如果操作A HB 操作B，那么A的执行结果对B一定可见，且A的执行顺序在B之前
- JMM允许编译器和CPU做指令重排序来优化性能，但重排序不能破坏HB规则，开发者只要遵守HB规则，就不用关心底层到底怎么重排的，JVM会保证正确性
- 核心规则一共8条：
1. 程序顺序规则：同一个线程内，前面的操作HB后面的操作，单线程里代码按写的顺序执行，这是最基础的规则
2. 监视器锁规则：对一个锁的unlock操作HB后续的同一个锁的lock操作，释放锁之前的所有修改，加锁后的线程都能看到
3. volatile规则：对volatile变量的写操作HB后续对这个变量的读操作，写入volatile的值，其他线程读的时候一定是最新的
4. 线程启动规则：Thread.start（）调用HB被启动线程里的所有操作，主线程在start（）之前做的事情，新线程都能看到
5. 线程终止规则：线程里的所有操作HB其他线程调用该线程的join（）返回，join返回后，能看到被等待线程的所有执行结果
6. 线程中断规则：interrupt（）调用HB被中断线程监测到中断事件，调用interrupt（）之前的操作，被中断线程通过isInterrupt（）检测到中断后都能看到
7. 对象终结原则：对象构造函数执行完毕HB finalize（）方法的开始，构造器里设置的字段值，finalize（）里一定能看到
8. 传递性规则：如果A HB B，B HB C，那么A HB C，这条规则让HB关系可以传递推导

什么是 Java 中的指令重排？
- 指令重排是编译器和CPU为了提升执行效率，在不改变单线程执行结果的前提下，对指令执行顺序做的调整，单线程下没问题，但多线程环境下可能有bug
- 指令重排发生在三个层面：
1. 编译器重排：Java编译器在生成字节码时，会根据优化策略调整代码顺序，比如把两条不相关的赋值语句调换位置，让寄存器分配更高效
2. CPU重排：现代CPU都是乱序执行的，内存有多条流水线并行处理指令，没有数据依赖的指令会被并行执行，完成后再按原始顺序提交结果
3. 内存系统重排：CPU有store buffer和invalidate queue，写操作可能先进buffer再刷到缓存，读操作可能从queue里拿到旧数据，从其他CPU视角看，内存操作的顺序和实际执行的顺序不一样
- JVM通过 as-if-serial 语义来保证单线程结果不变：不管怎么重排，单线程执行结果和按代码顺序执行的结果一样，但多线程下，JVM提供了 volatile，synchronized和HB规则来限制重排序，让开发者能控制操作的可见顺序

Java 中 volatile 关键字的作用是什么？
- volatile的核心作用就两个：可见性和禁止指令重排
1. 可见性：一个线程改了volatile变量，其他线程立马能看到最新值，没有volatile的话，线程各自在自己的CPU缓存里操作，改了也不一定同步到主内存，别的线程可能永远读到的都是旧值
2. 禁止指令重排：编译器和CPU为了性能会重排指令，单线程没问题，多线程就可能出幺蛾子，volatile通过内存屏障把重排限制住，写操作前插StoreStore屏障，读操作后插LoadLoad屏障，保证代码执行顺序符合预期
``` java
private static boolean flag = false;
public static void main(String[] args) {
    new Thread(() -> {
        while(!flag) {
            // 空转
        }
        System.out.println("Thread terminated");
    }).start();
    Thread.sleep(1000);
    flag = true; // 没有 volatile，这个改变对其他线程不可见
}
```
- 上面代码，子线程可能永远不会退出循环，因为子线程把flag缓存到自己的工作内存里了，主线程改了flag，子线程压根不知道，加上volatile之后，每次读flag都从主内存拿，改了立马能看到

Java 中的 final 关键字是否能保证变量的可见性？
- 不可以
- final保证的是final字段在构造方法里初始化完成后，只要构造期间没把this引用泄露出去，其他线程拿到对象引用时就一定能看到final字段的正确值
- 普通字段就没这待遇，构造函数里的写操作可能被重排序到构造函数外面，别的线程拿到对象时，字段值可能还没写进去

什么是 Java 中的 ABA 问题？
- ABA问题是CAS操作的一个坑：线程读到变量值是A，准备CAS更新的时候，发现值还是A，就以为没人动过，但实际上可能被别的线程改成B又改回A了，中间状态全丢
- 因为CAS本质就是比较并交换，它只看当前值和预期值是否相等，相等就更新，不想等就失败重试，它只比较值相不相等，不关心中间发生了什么

什么是 Java 中的线程同步？
- 线程同步就是给共享资源加把锁，保证同一时刻只有一个线程能访问它
- 可以想象成公共厕所的隔间：共享资源是那个唯一的🚽，线程是排队的人，同步机制就是门上的那把锁，没有锁的话，A正在用，B直接冲进去了，这就乱套了，有了锁，A进去后把门锁上，B在外面等，直到A出来释放锁，B才能进去
- 映射到业务中，多线程环境下，如果大家只读数据没事，但是多个线程同时修改同一个数据，比如抢最后一张票，由于CPU切换的太快，可能出现竞态条件
- 比如库存只剩一个，线程A读到了1，还没来得及减，线程B也读到了1，结果两个人都以为自己抢到了，库存变成了-1，这就是生产事故，同步就是为了避免这种超卖或脏数据

Java 中的线程安全是什么意思？
- 线程安全就是多个线程同时访问同一段代码或数据时，不管运行时环境怎么调度这些线程，程序都能表现出正确的结果，不会出现数据不一致或逻辑错乱
- 比如有个计数器count = 0， 两个线程同时执行count++，各执行100次，线程安全的情况下，结果一定是200；线程不安全的话可能是150，180这种乱七八糟的数字，因为count++本身不是原子操作，包含读取，加1，写回三步，两个线程可能同时读到旧值，然后各自加1写回，结果就少了
- 线程安全的计数器执行流程，线程A和B通过加锁机制串行访问count变量，每次只有一个线程能执行count++操作
- 线程不安全的计数器问题：线程A写回count = 6， 线程B也写回count = 6，实际上执行了两次自增，但结果只加了1
- 四种思路实现线程安全：
1. 互斥同步： 用synchronized 或 ReentrantLock 给临界区加锁，同一时刻只让一个线程进入，简单粗暴但有性能开销
2. 线程隔离： 用ThreadLocal让每个线程持有自己的变量副本，压根就不共享，自然不会冲突，数据库连接池常用这招，每个线程从池里拿连接时绑定到自己的ThreadLocal
3. 不可变对象： 用final修饰字段，或者像String那样设计成不可变，数据只能读不能改
4. 原子操作： 用AtomicInteger，AtomicLong这类原子类，底层靠CAS指令实现无锁的原子更新，比加锁轻量，适合简单的技术场景

什么是协程？Java 支持协程吗？
- 协程是用户态的轻量级线程，不由操作系统内核调度，而是由程序自己在用户空间里管理，创建和切换成本极低，内存占用只有几KB，不像线程那样动辄1MB
- Java从21开始正式支持协程，叫做虚拟线程，是Project Loom的成果
- 线程想像成公司的正式员工：
1. 成本高 ：每个人都要占工位（栈内存大，约1MB），入职离职手续麻烦（创建销毁慢）
2. 归老板管：OS就是老板，老板让你干你就干，让你停就停（抢占式调度），切换任务时还要开会交接（上下文切换开销大）
- 协程就像是外包临时工（日常实习生）
1. 成本极低： 自带小板凳就能干活（栈内存极小，几KB），招之即来挥之即去
2. 归项目组管：程序自己就是组长（用户态调度），组长发现A临时工在等快递（IO阻塞），立马让他去旁边休息，换B临时工上来干活，不需要惊动大老板（OS）
- 所以在同样的服务器配置下，你可能只能养1000个 “正式员工”（线程），但可以养100w个“临时工”（协程），这就解决了高并发下的资源瓶颈问题

线程的生命周期在 Java 中是如何定义的？
- Java线程的生命周期定义在ThreadState枚举，一共6种状态
1. NEW：线程对象刚new出来，还没调用start方法
2. RUNNABLE： 调用start后进入这个状态，这里包含了操作系统层面的就绪和运行两个状态，只要没被阻塞就算Runnable
3. BLOCKED： 线程想拿synchronized锁但没拿到，被挡在同步块外面等着
4. WAITING：调用了Object.wait, Thread.join或LockSupport.park,进入无限等待，必须被别的线程显式唤醒
5. TIMED_WAITING：和WAITING类似，但是带超时时间，比如Thread.sleep，Object.wait(timeout),Thread.join(timeout)
6. TERMINATED: run方法执行完毕或抛了未捕获异常，线程挂了
- 状态转换流程：NEW状态调用start方法进入RUNNABLE，RUNNABLE状态如果抢synchronized锁失败进入BLOCKED，拿到锁回到RUNNABLE，RUNNABLE状态调用sleep/wait（timeout）/join（timeout）进入TIMED_WAITING，超时或唤醒回到RUNNABLE，RUNNABLE状态run方法结束或异常进入TERMINATED

Java 中线程之间如何进行通信？
- Java线程通信本质就是让多个线程能够协调干活，分为共享内存和消息传递
- 共享内存是最直接的方式，多个线程读写同一块数据，但光能读写还不够，还得解决可见性，原子性，有序性。volatile关键字能保证可见性，但搞不定复合操作的原子性；synchronized和Lock能同时解决这三个问题，但代价是线程要排队
- 消息传递则是通过wait/notify，Condition，BlockingQueue这些机制，让线程之间互相打招呼，生产者干完活喊一声 “有活了” ，消费者听到了就去取； 消费者发现没活了就等着，直到生产者通知
- 最常用的通信方式：
1. wait/notify机制：配合synchronized使用，线程调wait释放锁进入等待，其他线程调notify唤醒
2. Lock + Condition： ReentrantLock的new Condition可以创建多个等待队列，比wait/notify灵活
3. BlockingQueue：生产者消费者模型的首选，put和take方法自带阻塞逻辑
4. volatile：轻量级同步，适合一个线程写，多个线程读的场景

Java 中如何创建多线程？
1. 实现Runnable接口，这是最基础的方式，实现run（）方法，然后把Runnable对象传给Thread构造函数，调用start（）启动线程
``` java
Thread thread = new Thread(() -> {
    System.out.println("线程执行中");
})
thread.start();
```
2. 继承Thread类，重写run（）方法，这种方式用的越来越少，因为Java只能单继承，继承 Thread就意味着没法继承别的类了
3. 使用Callable和FutureTask，跟Runnable的区别是能返回结果，也能抛异常
``` java
FutureTask<Integer> task = new FutureTask<>(() -> {
    return 100;
});
new Thread(task).start();
Integer result = task.get();
```
4. 使用线程池ExecutorService，这是生产环境最推荐的方式，线程池能复用线程，避免频繁创建销毁的开销
``` java
ExecutorService executor = Executors.newFixedThreadPool(10);
executor.submit(() -> System.out.println("任务执行"));
```
5. 使用CompletableFuture，Java8引入的异步编程工具，底层默认用ForkJoinPool，能很方便的组合多个异步任务，处理任务之间的依赖关系
``` java
CompletableFuture.runAsync(() -> System.out.println("异步执行"));
```
- 这5种方式可以分为两类：直接创建线程和通过线程池，前三种是直接创建，后两种走线程池

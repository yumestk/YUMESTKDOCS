Java 的 synchronized 是怎么实现的？
- synchronized 的实现原理依赖于JVM的Monitor监视器锁和对象头Mark Word
- 修饰方法和代码块的实现方式略有不同：
1. synchronized修饰方法时，会在方法的访问标志中加上ACC——SYNCHRONIZED标志，线程进入方法前，JVM检查这个标志，有的话就得先拿到对象的监视器锁才能执行
2. synchronized修饰代码块时，编译器在代码块前后插入monitorenter和monitorexit字节码指令，monitorenter是加锁，monitorexit就是解锁，而且编译器会自动生成异常处理的monitorexit，保证抛异常也能正常解锁，synchronized代码块编译后会在代码块入口插入monitorenter指令，在正常出口和异常出口各插入一条monitorexit，线程执行monitorenter时尝试获取对象的monitorenter时尝试获取对象的monitor锁，成功则进入临界区，失败则阻塞等待，执行monitorexit时释放锁并唤醒等待队列中的线程

Synchronized 修饰静态方法和修饰普通方法有什么区别？
- 核心区别就一句话：锁的对象不一样
- 修饰静态方法时，锁的是这个类的Class对象，不管你new了多少个实例，它们公用同一把锁，因为一个类的Class对象在JVM里只有一份
- 修饰普通方法时，锁的是this，也就是当前调用这个方法的对象实例，每个对象实例都有自己独立的锁，A对象和B对象互不影响
- 这就导致了一个关键差异，两个线程分别调用同一个对象的静态同步方法和实例同步方法，是不会互斥的，因为锁的根本不是一个东西

Java 中的 synchronized 轻量级锁是否会进行自旋？
- 轻量级锁CAS失败之后不会自旋，而是直接膨胀成重量级锁
- 轻量级锁加锁流程：线程在栈帧中创建Lock Record，然后尝试用CAS把对象头的Mark Word替换为指向Lock Record的指针，CAS成功就拿到锁直接返回；CAS失败说明有竞争，直接走锁膨胀逻辑升级成重量级锁，中间没有任何自旋操作
- 真正有自旋的地方是重量级锁阶段，当线程竞争重量级锁失败时，会先自旋一段时间等待锁释放，而不是立刻阻塞，毕竟线程阻塞入队再唤醒的开销还是挺大的，如果持锁线程很快就释放了，自旋一下就能拿到锁，省去了上下文切换的成本

Synchronized 能不能禁止指令重排序？
- 不能，它跟volatile的作用机制完全不同
- synchronized保证的是有序性，但这里的有序指的是通过加锁把并行变串行实现的，在同一时刻，只有一个线程能进入sunchronized代码块，其他线程都得在外面等，这样一来，多线程之间看到的操作顺序就是确定的，但是synchronized代码块内部该怎么重排还是怎么重排，JVM和CPU只要保证单线程执行结果不变，就可以随意调整指令顺序

当 Java 的 synchronized 升级到重量级锁后，所有线程都释放锁了，此时它还是重量级锁吗？
- 要分两步来看，当重量级锁释放之后（所有线程退出同步块），对象头并不会立即变成无锁状态，此时 mark word 仍然指向Object Monitor，锁依旧处于膨胀（inflated）状态
- 但是当JVM后台的monitor deflation（锁收缩）机制运行后，空闲的Object Monitor会被回收，对象头会恢复为无锁状态
- 当deflation完成后，如果有新线程来加锁，会从轻量级锁重新开始，而不是直接走重量级锁流程
- 锁释放=恢复无锁和一旦膨胀就永远是重量级锁这两种理解都不准确
- 释放锁不等于deflation，释放锁只是把Object Monitor的owner字段清空，对象头不变
- deflation是JVM后台独立执行的操作，会将闲置的Object Monitor回收，恢复对象头为无锁状态
- deflation之后，JVM不会记住这个对象曾经膨胀过，新线程来加锁时会根据当前竞争情况重新判断
- 总结：1.释放锁(monitorexit) -> owner=NULL, mark word不变，仍是重量级锁 2.JVM Deflation -> 回收 ObjectMonitor，mark word恢复，变为无锁 3.新线程加锁 -> 根据竞争情况重新判断，从轻量级锁开始

什么是 Java 中的锁自适应自旋？
- 自适应自旋是synchronized重量级锁的一个优化手段：线程竞争锁失败时不会立刻阻塞，而是先自旋一会儿等锁释放，自旋的时间不是写死的，JVM会根据上次自旋的结果动态调整
- 具体逻辑：线程抢锁失败后，不是立刻挂起，而是原地空转几圈看看能不能释放，如果上次自旋很快就拿到锁了，JVM认为这次自旋也很可能成功，就允许多转几圈；若果上次白转了半天最后还是阻塞了，JVM就认为这锁可能竞争比较激烈，这次少转甚至不转，直接阻塞
- 自旋的原因是：线程阻塞和唤醒的成本很高，要涉及用户态和内核态的切换，如果持锁线程很快就释放了，自旋一下就能拿到锁，省去了上下文切换的开销，但自旋也不是没代价，CPU在空转，所以需要自适应机制来平衡

Java 中 ReentrantLock 的实现原理是什么？
- ReentrantLock底层就是AQS+CAS那套东西，核心是一个int类型的state变量和一个等待队列
- 加锁的时候，线程用CAS尝试把state从0改成1，改成功了就拿到锁，把exclusiveOwnerThread设成自己，改失败了说明锁被别人占着，乖乖去同步队列尾部排队，然后park挂起
- 解锁的时候，state减1，如果减到0就唤醒队列里的第一个等待线程
- 可重入的实现很简单，同一个线程再次加锁，发现exclusiveOwnerThread就是自己，直接把state加1，不用排队，释放的时候每次减1，减到0才真正释放锁
- 公平锁和非公平锁的区别就在于：非公平锁上来就CAS抢一把，抢到了就直接用，抢不到再排队；公平锁老老实实先检查队列里有没有人在等，有人等就去排队，没人才CAS抢

你了解 Java 中的读写锁吗？
- 读写锁允许多个线程同时读取共享资源，但写操作时只能有一个线程独占，读读不互斥，读写互斥，写写也互斥，这种机制适合读多写少的场景，比如配置中心，缓存系统这类读操作占90%以上的业务
- Java中的ReadWriteLock接口由ReentrantReadWriteLock实现，提供两种锁模式：
1. 读锁是共享锁，多个线程可以同时持有，前提是没有线程正在持有写锁
2. 写锁是独占锁，持有写锁时别的线程既拿不到写锁也拿不到读锁，保证写操作的原子性
- 读写锁的核心机制：
1. 读锁获取：检查是否有写锁被持有，没有则可以获取，读锁计数加1
2. 写锁获取：检查是否有任何锁被持有，没有则可以获取，独占资源
3. 读读并行：多个线程同时持有读锁，并发读取
4. 读写互斥：有读锁时写锁要等待，有写锁时读锁也要等待

什么是 Java 的 StampedLock?
- StampedLock是Java8引入的一种锁机制，相比ReentrantReadWriteLock最大的改进是引入了乐观读锁，传统的读写锁在读的时候也要加锁，而StampedLock的乐观锁压根不加锁，只是拿一个时间戳，读完再校验一下这期间有没有写操作，如果没有，读就成了；如果有写操作介入，在降级为悲观读锁重试
- 这种设计在读多写少的场景下性能提升非常明显，因为大部分读操作都不用真正竞争锁
- StampedLockd提供三种锁模式：
1. 写锁：独占模式，和ReentrantLock一样同一时刻只能有一个线程持有写锁
2. 悲观读锁：共享模式，多个线程可以同时持有读锁，但写锁必须等所有读锁释放
3. 乐观读锁：不是真正的锁，只是获取一个stamp时间戳，读完后通过validate方法校验期间有没有写操作

Synchronized 和 ReentrantLock 有什么区别？
- synchronized是Java内置的关键字，用起来最简单，锁的释放和获取由JVM自动管理，不用操心忘记解锁的问题，ReentrantLock是JDK1.5引入的显式锁，功能更丰富，但必须手动lock/unlock，一般配合try-finally使用

| 特性         | synchronized         | ReentrantLock                              |
| ------------ | -------------------- | ------------------------------------------ |
| 锁类型       | JVM 内置，隐式锁     | JUC 类库，显式锁                           |
| 获取/释放    | 自动                 | 手动，必须在 finally 中 unlock             |
| 公平性       | 只支持非公平         | 支持公平/非公平                            |
| 可中断       | 不支持               | lockInterruptibly 可响应中断               |
| 超时获取     | 不支持               | tryLock(timeout) 避免死锁                  |
| 条件变量     | 只有一个隐式等待队列 | 可创建多个 Condition                       |
| 可重入       | 支持                 | 支持                                       |

Volatile 与 Synchronized 的区别是什么？
- 这俩都能保证可见性，但volatile不保证原子性，synchronized两者都保证
- volatile只能修饰变量，作用是告诉JVM这个变量可能被多个线程同时访问，每次读都要从主内存拿到最新值，每次写都要立刻刷回主内存，它不加锁，开销很小，但i++这种 “读-改-写” 的复合操作它保护不了，两个线程可能同时读到同一个值，各自加1后写回去，结果只加了1
- synchronized可以修饰方法或代码块，进入时加锁，退出时解锁，同一时刻只有一个线程能持有锁，其他线程只能排队等着，它能保证原子性，也能保证可见性，因为加锁时会清空工作内存从主内存重新读，解锁时会把修改刷回主内存，代价就是锁竞争会导致线程阻塞，性能开销比volatil大得多

如何优化 Java 中的锁的使用？
- 锁优化的核心思路就两条：能不锁就不锁，非要锁就锁得越短越好
1. 缩小锁的粒度和持有时间：把锁的范围收到最小，只在真正需要同步的那几行代码上加锁，比如一个方法里只有3行涉及共享变量，就别把整个方法都synchronized了，只锁那3行就够，还有一招是把一把大锁拆成多把小锁，ConcurrentHashMap就是典型，JDK7用了16个Segment，每个Segment一把锁，16个线程可以同时写不同的Segment；JDK8更激进，直接锁到单个桶的粒度，并发度拉满
2. 读写分离：读多写少的场景用ReentrantReadWriteLock，读锁可以共享，10个线程同时读没问题，只有写的时候才独占，像配置缓存这种场景，99%是读操作，用读写性能比synchronized高一个量级
3. 干脆别用锁：CAS和原子类能搞定就不用锁，AtomicInteger的incrementAndGet底层就是一条CPU的cmpxchg指令，没有线程挂起和唤醒的开销，QPS能从几万飙到几十万

什么是 Java 的 CAS (Compare-And-Swap) 操作？
- CAS是一种硬件级别的原子操作，它比较内存中的某个值是否为预期值，如果是就更新为新值，不是就不动，整个比较和交换的过程是一条CPU指令完成的，不可能被打断
- CAS需要三个操作数：变量的内存地址，就的预期值，新值，CPU根据内存地址拿到当前值，和预期值比较，相等就替换为新值，不相等就返回失败，失败了一般会自旋重试，直到成功为止

``` mermaid
flowchart LR
    A[读取当前值] --> B{当前值 == 预期值?}
    B -- 是 --> C[更新为新值]
    C --> D[返回成功]
    B -- 否 --> E[返回失败]
    E --> F[自旋重试]
    F --> A
```

> [!NOTE]
> 抢不到锁不死心，原地循环疯狂重试 = 自旋
> 一、Mark Word（标记字）
> 存在于 Java 对象头里，占 64 位 / 32 位，存对象核心状态信息
> 存的东西
> 哈希码 hashcode
> GC 分代年龄（0-15）
>1. 锁状态标志（无锁 / 偏向锁 / 轻量锁 / 重量锁）
>2. 偏向锁线程 ID
>3. 自旋次数
> 4. 作用
>JVM 靠它识别对象处于哪种锁状态，实现锁升级、偏向锁、轻量级锁逻辑。
>二、Owner（锁持有者）
>就是：当前这把锁「被哪个线程持有」
>偏向锁 Owner : Mark Word 里存线程 ID = 这个线程独占偏向锁
>轻量级锁 Owner : 栈帧锁记录指针指向对象头，代表当前线程持有
>重量级锁 Owner : 底层 Monitor 监视器里记录持有锁的线程
>直白理解
>Owner = 锁的主人线程
>有 Owner：锁被占了
>无 Owner：空闲可抢

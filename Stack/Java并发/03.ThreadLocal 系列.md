为什么在 Java 中需要使用 ThreadLocal?
- ThreadLocal 解决的问题是线程隔离，让每个线程都有自己独立的变量副本，各玩各的，不需要加锁也不会有竞争，以空间换时间，无锁，高性能
- 跟传统的synchronized，Lock相比，ThreadLocal走的是完全不同的思路，加锁是让多个线程排队访问同一份数据，ThreadLocal是让每个线程都有自己的一份数据，不存在共享就没有竞争，没有竞争就不用加锁，性能就上去了
- 上下文传递时也用到了，代码层级深的时候，比如controller->service->dao这种调用链，如果每一层都用到用户ID，事务对象这些东西，一层层传参写起来很烦，用ThreadLocal存一下，哪层需要就直接get，代码干净很多
- Spring的事务管理就是这么干的，TransactionSynchronizationManagere用ThreadLocal存当前线程的数据库连接和事务状态，你在Service层开了事务，DAO层自己就能拿到同一个连接，不用手动传来传去

Java 中的 ThreadLocal 是如何实现线程资源隔离的？
- ThreadLocal 实现线程隔离的关键在于：数据不是存在ThreadLocal对象里，而是存在每个Thread对象自己的Map里
- 每个Thread对象内部有一个threadLocals字段，类型是ThreadLocalMap，当你调用threadLocal.get() 的时候，他先拿到当前线程自身作为key去查，所以同一个ThreadLocal对象，线程A调用get查的是A的Map，线程B调用get查的是B的Map，各找各的，自然就隔离了

为什么 Java 中的 ThreadLocal 对 key 的引用为弱引用？
- ThreadLocalMap中Entry对key使用弱引用，核心目的是配合线程池场景下的内存回收，线程池里的线程可能存活整个应用生命周期，如果Entry对ThreadLocal是强引用，那即使业务代码已经不用这个ThreadLocal对象，它也没法被回收，因为线程一直持有它
- GC场景说明：当栈上的强引用消失时，由于Entry对Key （ThreadLocal） 只有弱引用，下一次GC会回收Key，但Value是强引用，若线程不结束，Value无法被回收，导致内存泄漏（Key为null，Value有值）
- 弱引用的特性是：只要发生GC，不管内存够不够，只被弱引用指向的对象都会被回收，所以当业务代码执行完，栈帧出栈，ThreadLocal的强引用没了，剩下的弱引用在GC时就会被干掉

Java 中使用 ThreadLocal 的最佳实践是什么？
- ThreadLocal的最佳实践可以总结为：**用完就清**
1. 声明成static final， 避免每次new一个ThreadLocal对象，一个ThreadLocal实例对应一个线程变量就够了，没必要搞多个实例
2. 用withInitial设置默认值，防止get时拿到null
3. 业务逻辑执行完必须调remove，尤其是在Web项目里，Tomcat的工作线程是复用的，不remove的话下一个请求可能拿到上一个请求的数据
4. 能用方法参数传递的上下文信息，就别用ThreadLocal，ThreadLocal本质上是隐式传参，滥用会让代码难以追踪和调试

Java 中的 InheritableThreadLocal 是什么？
- InheritableThreadLocal是ThreadLocal的子类，解决的是父子线程之间值传递的问题，普通ThreadLocal每个线程的数据是完全隔离的，父线程设置的值子线程拿不到，InheritableThreadLocal在创建子线程时会把父线程的值拷贝一份给子线程
- 工作机制：Thread类里除了threadLocals还有个inheritableThreadLocal字段，创建子线程时，构造函数会检查父线程的inheritableThreadLocal是否有值，有的话就浅拷贝Map，复制引用，遍历复制到子线程的iTL里，这个拷贝只发生在new Thread的时候，之后父线程再怎么改，子线程都不受影响

ThreadLocal 的缺点？
- ThreadLocal在正常使用下问题不大，但在线程池场景或数据量大的情况下，有三个主要缺点，内存泄漏风险，哈希冲突效率低，清理开销不可控
- 内存泄漏是最常见的，ThreadLocalMap的key是Weak Reference包装的ThreadLoca对象，value是强引用，当外部的ThreadLocal引用被置为null之后，key会被GC回收成null，但value还在，这块内存就泄漏了，线程池里线程会被复用，生命周期很长，泄漏的value一直堆积，最终可能OOM

为什么 Netty 不使用 ThreadLocal 而是自定义了一个 FastThreadLocal ?
- Netty自己搞了FastThreadLocal主要是为了解决ThreadLocal的两个性能问题：哈希冲突的线性探测效率低和弱引用带来的内存泄漏风险
- 核心思路是用空间换时间，彻底干掉哈希冲突，每个FastThreadLocal在构造时就分配一个全局唯一的index，存取数据直接用这个index作为数组下标，O（1）定位，压根不需要算哈希也不会有冲突
- 要改ThreadLocalMap就得动Thread，但JDK的Thread改不了，所以Netty配套搞了三个类：FastThreadLocal对标ThreadLocal， InternalThreadLocalMap对标ThreadLocalMap，FastThreadLocalThread对标Thread，三者配合使用才能发挥最大性能

什么是 Java 的 TransmittableThreadLocal?
- TTL是阿里开源的一个工具类，专门解决线程池场景下ThreadLocal传递失效的问题
- 普通ThreadLocal只能在当前线程内使用，InheritableThreadLocal可以传给子线程，但只在new Thread的时候生效，线程池的线程是预创建好的，任务提交时线程早就存在了，InheritableThreadLocal压根不起作用
- TTL的核心思路是任务提交时把主线程的ThreadLocal快照保存到任务对象里，线程池的线程执行任务前把快照恢复到自己的上下文，执行完再清理还原，整个过程三步：Capture（拷贝快照）-> Replay（重放快照）-> Restore（复原上下文）
1. 主线程调用ttl.set（）设置值，同时TTL对象被记录到holder里
2. 提交任务时，TtlRunnable构造函数遍历holder拷贝所有TTL的值作为快照
3. 线程池的线程执行任务前，先备份自己当前的上下文，再把快照恢复进去
4. 任务执行完毕后，把备份的上下文还原回去，保证线程干净


> [!NOTE]
> 强引用：正常业务对象、变量，必须常驻不能随便丢
> 弱引用：缓存、临时数据、图片缓存，内存不够可以随时丢，不影响业务,GC一来就分手

HTTP 1.0 和 2.0 有什么区别？
- HTTP的几个版本，其实就是围绕两个核心痛点演进的：连接成本（TCP链接多，慢）；头部冗余，队头阻塞，传输效率低
- HTTP1.0到1.1的核心升级，解决连接成本问题：最大的区别就是长连接（  Keep-Alive），HTTP1.0是短命的，每次请求都要走三次握手，四次挥手，效率低，HTTP1.1默认开启长连接，同一个TCP连接可以复用，不用频繁建连了；除此之外，在1.0时期一台服务器只能部署一个网站，1.1引入Host头，允许一台物理机部署多个虚拟主机（域名），这是现代云主机的基础
- HTTP1.1到2.0的核心革命，解决队头阻塞与带宽问题：
1. 多路复用：HTTP1.1虽然有长连接，但请求是串行，如果前面的请求卡住了，后面都得等，这叫队头阻塞，HTTP2.0引入了流（stream）的概念，他把一个TCP连接拆成了无数个逻辑上的流，每个请求打散成二进制帧乱序发送，到了服务端再组装，这就好比1.1是单车道，前面的车坏了后面全堵死；2.0是多车道高速公路，互不干扰
2. 二进制分帧：1.x是纯文本协议（肉眼可读），解析慢且容易出错，2.0全面改为二进制格式，机器解析更快更健壮
3. 头部压缩（HPACK）：1.1的Header往往很大（Cookie，User-Agent），而且每次请求都要重复发，浪费带宽，2.0在客户端和服务器端各维护一张索引表，重复的Header只需要发一个索引ID，大大减少了传输量
4. 服务器推送： 服务端可以预判你的需求，比如你请求了index.html，服务端知道你肯定还要style.css，就主动推给你，不用你再发请求来要（不过不好控制，Chrome等主流浏览器已经废弃了对它的支持）

HTTP 2.0 和 3.0 有什么区别？
- HTTP/2 和 HTTP/3 最大的区别，不在HTTP本身，而底层的传输层变了，从TCP变成了UDP
- 有一个通俗的比喻，HTTP/2是把100个人塞进一辆公交车，还得排队上车； HTTP/3是给每个人发了辆车，谁想走就走，谁丢了包只重发自己的车
- 核心区别：
1. 基于的传输层协议不同，解决队头阻塞，建连速度，连接迁移，头部压缩算法升级

常见的 HTTP 状态码有哪些？
- 1xx信息响应： 100 Continue表示服务器收到了请求的初步部分，客户端可以继续发送请求体； 101 Switching Protocols表示服务器同意切换协议，比如HTTP升级到WebSocket
- 2xx成功： 200OK是最常见的，表示请求成功； 201 Created 表示资源创建成功，Post请求创建新用户或新订单就返回这个；204 No Content 表示请求成功但没有响应体，DELETE 删除资源后通常返回这个
- 3xx 重定向： 301 Moved Permanently 表示资源永久搬家了，浏览器会自动更新书签，搜索引擎也会更新索引； 302 Found 表示临时重定向，资源暂时在别的地方，下次还是要访问原来的地址； 304 Not Modified 配合缓存使用，表示资源没变过，直接用本地缓存就行，省的再传一遍
- 4xx 客户端错误： 400 Bad Request 说明请求格式有问题，参数不对或JSON格式错了。 401 Unauthorized 表示没登陆或者 Token 过期了，需要先认证； 403 Forbidden 表示认证过了但是没权限访问，普通用户访问管理员接口就会返回这个。404 Not Found 表示资源不存在，URL写错了或者资源被删了
- 5xx 服务器错误：500 Internal Server Error 是服务端代码抛异常了，空指针，数据库连接失败这些问题都可能导致500. 502 Bad Gateway说明网关或代理从上游拿到了无效响应，上游服务挂了或者响应格式不对； 503 Service Unavailable 表示服务器暂时不可用，可能是在维护或流量太大扛不住了； 504 Gateway Timeout 是网关等上游响应超时了，上游服务处理太慢

HTTP 请求包含哪些内容，请求头和请求体有哪些类型？
- HTTP请求由四部分组成：请求行、请求头、空行、请求体
- 请求行包含三个东西：请求方法、请求路径、协议版本。比如Get/api/users HTTP/1.1，一眼就能看出来是用GET方法请求/api/users这个资源，协议是HTTP1.1
- 请求头是一堆键值对，告诉服务器客户端各种信息：我是什么浏览器，我能接受什么格式的响应，我带了什么认证信息
- 空行就是一个换行，用来分隔请求头和请求体，服务器解析的时候靠这个判断请求头就结束了
- 请求体在POST、PUT这类方法里才有，GET请求一般没有请求体，请求体里放的就是你要提交给服务器的数据
- 常见的请求头有这几类：
1. 通用头部：请求和响应都能用，比如Cache- Control控制缓存，Connection控制连接是否保持
2. 请求头部：只用于请求，比如Host指定目标主机、User-Agent标识客户端、Accept告诉服务器想要什么格式，Authorization带认证信息
3. 实体头部：描述请求体的，比如Content-Type说明请求体是什么格式、Content-Length说明请求体有多大
- 请求体常见格式：
1. application/x-www-form-urlencoded：最传统的表单提交格式，数据编码成key1=value&key2=value2这种形式
2. multipart/form-date：上传文件必须用这个，数据用分界符分割，每个部分可以有不同的类型
3. application/json：现在最常用的格式，RESTful API基本都用JSON传数据
4. text/plain：纯文本，很少用
5. application/xml：XML格式，老系统或者SOAP接口会用

HTTP 和 HTTPS 有什么区别？
- HTTP相当于在教室里传纸条，纸条上写什么，路过的每个人（路由器、运营商）都能看见，甚至还能偷偷改两笔，或者冒充别人给你传
- 而HTTPS则是把纸条放进了上了锁的保险箱，只有手持钥匙的收件人才能打开，既看不见也改不了
1. 安全性 （最本质的区别）
Http是裸奔的，数据明文传输（窃听，篡改，冒充）
而HTTPS是穿了防弹衣的，他在HTTP下加入了一个SSL/TLS协议。通过加密解决了窃听，通过校验解决了篡改，通过数字证书解决了冒充
2. HTTP简单粗暴，只要三次握手就能发数据
相比而言HTTPS确实会慢一点，因为TCP握手之后，还得多进行一次TLS握手（通常是4次交互）
3. 端口
HTTP默认80，HTTPS默认443

HTTP 中 GET 和 POST 的区别是什么？
    从HTTP规范的语义来看，GET用于获取资源，不改变服务器状态；POST用于提交数据，通常会创建或更新资源
1. 参数传递方式：
GET把参数拼在URL上，长度受浏览器和服务器限制，一般在2KB左右，POST把参数放在请求体里，理论上没有大小限制，更适合传大块数据
2. 安全性：
GET的参数直接暴露在URL里，会被浏览器历史记录、服务器日志、代理缓存记下来，不适合传密码这类敏感信息。POST把数据藏在请求体里，看起来安全一点，但本质上还是明文，还是得靠HTTPS
3. 幂等性：
按规范GET是幂等的，同一个请求发10遍结果都一样；POST不是幂等的，发10遍请求可能创建10条数据
4. 缓存机制：
GET请求能被浏览器和CDN缓存，适合图片、静态页面这类不常变的资源。POST请求默认不缓存，每次都会打到服务器

**WebSocket 与 HTTP 有什么区别？**
WebSocket和HTTP最本质的区别就在于通信方向。HTTP是单向的，客户端问一句，服务器答一句，答完就断；WebSocket是双向的，建立连接后，双方都能主动给对方发消息，想什么时候发就什么时候发，不用等对方先开口。
HTTP的问题在于实时性太差，比如做一个在线聊天室，用HTTP只能靠客户端不断轮询，每隔1秒发请求问有无新消息，服务器每次都要回应
WebSocket就是为了解决这个痛点，它复用了HTTP的握手流程，第一次请求还是走HTTP，带上Upgrade；websocket头告诉服务器要升级协议，服务器同意后返回101状态码，之后整个TCP连接就切到WebSocket协议了，双方可以随时推送数据，连接一直保持，省掉了反复建连的开销

服务端是如何解析 HTTP 请求的数据？
服务端收到的HTTP请求就是一段文本，格式是固定的：请求行+请求头+空行+请求体。服务端按这个格式逐行解析就行
一个典型的HTTP请求：
``` text
POST /api/login HTTP/1.1
Host: www.mianshiya.com
Content-Type: application/json
Content-Length: 42

{"username":"admin","password":"123456"}
```

解析过程：
1. 先读第一行请求行，按空格切分得到请求方法POST、URL /api/login、协议版本 HTTP/1.1
2. 接着逐行读请求头，每行都是key：value格式，遇到空行就停。Host告诉服务器请求的是哪个域名，Content-Type告诉请求体是什么格式，Content-Length告诉请求体有多少字节
3. 空行之后就是请求体了。服务端根据Content-Length读指定长度的数据，再根据Content-Type决定怎么解析。application/json就用JSON解析器，application/x-www-form-urlencoded就按&切分键值对
4. 解析完成后，根据URL路由到对应的处理逻辑，执行业务代码
5. 处理完生成响应，响应格式也是固定的：状态行+响应头+空行+响应体
``` text
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 27

{"code":0,"msg":"success"}
```

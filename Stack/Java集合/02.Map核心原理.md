说说 Java 中 HashMap 的原理？
- HashMap底层就是个数组,结合链表＋红黑树来解决冲突,HashMap是非线程安全的,多线程环境下可能出问题:比如数据覆盖或者死循环,这时候就是ConcurrentHashMap登场了
- 怎么存:存一个Key-Value时,先算Key的hashcode, 然后用(table.length - 1) & hash决定存到数组的哪个下标位置(哪个桶)
- 冲突了怎么办: 两个不同的Key算出来下标一样就叫哈希冲突,HashMap的解决办法就是用链表给串起来,如果同一个下标下的链表超过8个节点且数组长度大于等于64,那么链表就转为红黑树,查找速度从O(n)优化为O(logn)
- 扩容机制: 负载因子默认0.75,数组用了75％就自动扩容,把数组大小翻倍,然后把所有数据rehash到新数组里,但是比较耗性能,所以最好一开始就给个预估容量
- HashMap底层是数组加链表,Java8引入红黑树优化,存取时通过hashcode确定下标,遇到冲突就用链表或者红黑树解决,为保证性能,达到阈值会扩容

什么是 Hash 碰撞？怎么解决哈希碰撞？
- Hash碰撞就是两个不同的Key经过hash函数计算后落到了同一个数组下标
- 拉链法:把每个槽变成一个链表
- 开放寻址法:寻找下一个可用位置,ThreadLocalMap就是用的这个
- 再哈希法: 使用第二个哈希函数计算新的索引位置,直到无冲突

Java 中 HashMap 的扩容机制是怎样的？
- 由负载因子控制,扩容就是容量直接翻倍,始终保持2的幂次,扩容后所有元素要重新分配位置,这个过程叫rehashing,jdk1.7之前要一个个重新取模算下标,1.8做了优化:元素只会在原位置不变或者原位置加旧容量;所以扩容时链表中的元素可能只需要部分移动,只看hash的第5位是0还是1(位运算判断高位)

为什么 HashMap 在 Java 中扩容时采用 2 的 n 次方倍？
- 位运算替代取模和扩容时高效迁移,位运算公式是(n - 1) & hash
- 只有当n是2的次方时,n - 1的二进制低位才全是1,这样&运算能让hash值的低位完全保留,保证索引均匀分布;比如n - 1 = 15,hash值跟它&一下,低4位原封不动拿来当下标

为什么 Java 中 HashMap 的默认负载因子是 0.75？
- 0.75是时间和空间之间的平衡点,负载因子太低,扩容太频繁,内存浪费;负载因子太高,冲突太多,查找变慢

为什么 JDK 1.8 对 HashMap 进行了红黑树的改动？
- 为了解决哈希冲突严重时查找性能退化的问题

JDK 1.8 对 HashMap 除了红黑树还进行了哪些改动？
- 哈希函数优化,1.7的hash函数做了四次异或运算,1.8只做一次,把Key的hashCode高16和低16位异或一下完事;
- 扩容不用挨个取模, 1.7扩容时每个元素都要重新算一遍hash%newCapacity,1.8要么留原位置,要么挪到原位置+老数组长度
- 头插法改尾插法,1.7插入新节点直接往链表头部塞,扩容时链表会倒过来,多线程情况下容易把链表搞成环,CPU直接跑满死循环,1.8改成尾插法,扩容后顺序不变,避免了成环问题

使用 HashMap 时，有哪些提升性能的技巧？
- 初始容量一定要给够,扩容的话都是O(n)的全量rehash,性能很差
- 负载因子别乱调,调低了冲突少查询快,但是内存占用会飙上去
- 确保hashCode均匀分布,不然查找效率会从O(1)退化到O(n)链表或O(logn)红黑树

Java 中的 HashMap 和 Hashtable 有什么区别？
- 最核心的区别就是线程安全性
- Hashtable的方法都加了synchronized,是线程安全的,但代价是性能差;HashMap没加锁,单线程下跑得快,但多线程环境得自己保证同步

Java 中的 HashSet 和 HashMap 有什么区别？
- HashSet和HashMap最本质的区别是存储模型不同,HashMap存的是键值对,一个Key对应一个value;HashSet只存单个元素,压根没有value的概念
- HashSet底层就是套了一层HashMap,你往HashSet里add一个元素,它内部调用的HashMap的put,把元素当Key,value统一塞一个叫PRESENT的空Object占位,所以HashSet的去重能力,查找效率,全是HashMap在干活

Java 中的 LinkedHashMap 是什么？
- LinkedHashMap就是在HashMap的基础上加了双向链表,把所有节点串起来

Java 中的 TreeMap 是什么？
- 底层红黑树,天生有序,它会根据key的大小自动排序,要么key实现Comparable接口走自然排序,要么构造时传个Comparator自定义排序规则
- 用的场景不算多,常用在签名加密,有些接口签名需要把参数按字母序排序再拼接,直接把参数塞到TreeMap,取出来就是有序的,省的自己排序

Java 中的 IdentityHashMap 是什么？
- IdentityHashMap是Java集合框架里一个特殊的Map实现,它用 "==" 判断键是否相等,而不是像HashMap那样用equals方法,只有两个键的内存地址完全一样,才会被认为是一个键
- 需要跟踪对象实例本身,而不是逻辑上相等的对象.比如框架里要给每一个代理对象挂一份元数据,就算两个代理背后的目标对象内容一样,也得分开记录
- 构建深拷贝或序列化工具时,需要检测对象图中的循环引用,用IdentityHashMap记录已经处理过的对象,能准确识别 "同一个对象出现了两次"

Java 中的 WeakHashMap 是什么？
- WeakHashMap是Java集合框架中一个特殊的Map实现,它的键用弱引用包装,当一个键对象在外部没有任何强引用指向它时,GC就可以回收这个键,随后WeakHashMap会自动删除对应的键值对
- 这种特性最典型的应用场景是内存敏感的缓存,比如你想缓存一些计算结果,但又不希望这个缓存阻止对象被回收,当缓存的Key在业务代码里不再被使用时,缓存条目会自动消失,不用手动清理
- WeakHashMap = 自动清理的缓存
没人用的 key，会自动被垃圾回收，不会造成内存泄漏。

ConcurrentHashMap 和 Hashtable 的区别是什么？
- 两者都是线程安全的哈希表,但锁机制完全不同,这直接导致了性能上的巨大差异
- Hashtable是JDK1.0的老古董,所有方法都加了synchronized,相当于给整个表加了一把大锁,10个线程同时来访问,9个得排队,并发度就是1,所有操作串行
- ConcurrentHashMap在JDK8采用CAS + synchronized配合的细粒度锁,往空桶里边查数据用CAS无锁写入,有冲突时才用synchronized锁住单个桶的头节点,一张表默认16个桶,理论上16个线程可以同时写不同的桶,互不干扰,每个桶一把锁,不同桶可并行,并发度等于桶数量
- Hashtable和ConcurrentHashMap不允许key或value为null,但是HashMap允许
- 锁粒度决定并发能力,现代高并发场景Hashtable基本没人用了

Java 中 ConcurrentHashMap 1.7 和 1.8 之间有哪些区别？
- 两个版本的核心区别在于锁的粒度
- JDK 1.7采用分段锁设计,底层把整个数组分成16个Segment,每个Segment里面是一个完整的HashMap加一把ReentranLock.不同线程访问不同Segment完全不挡,同一Segment才有锁竞争,并发度最高16
- JDK1.8移除了Segment,锁粒度细化到数组的每个槽位,数据结构跟HashMap同步,变成数组+链表+红黑树,插入时先用CAS无锁尝试插到数组位置,真冲突了才用synchronized,而且只锁链表或树的头节点,其他线程照样可以操作别的bucket,并发度大大增加

Java 中 ConcurrentHashMap 的 get 方法是否需要加锁？
- 不需要加锁,ConcurrentHashMap的get方法完全依赖volatile语义来保证线程安全,读取时能拿到最新的数据,不会有并发问题
- 具体实现上,数组元素的读取用的是`Unsafe#getObjectVolatile`, Node节点的val和next指针也被volatile修饰,这三个关键点保证了整个读取链路的可见性
- get流程分三步:1.通过hash定位数组下标,用getObjectValue读取数组槽位 2.如有hash冲突,沿着next指针遍历链表,next是volatile的
- 找到目标节点后读取val,val也是volatile的

为什么 Java 的 ConcurrentHashMap 不支持 key 或 value 为 
null？
- 核心原因是消除歧义,ConcurrentHashMap是为并发环境设计的,如果允许value为null,调用get(key)返回null时,你没法判断key不存在还是value本身就是null
- 单线程下这个问题可以用containsKey来区分,但在并发环境下,这两步操作之间可能有其他线程在修改map,导致判断失效
- 另一个原因是简化实现,如果允许null,put,get,containsKey这些方法都得加额外逻辑去处理空值情况,代码复杂度上升,出bug的概率也会增加

Java 中的 CopyOnWriteArrayList 是什么？
- CopyOnWriteArrayList是java.util.concurrent包下的一个线程安全的List实现,核心机制是写时复制: 每次写操作都要复制一份新数组,在新数组上修改,然后替换旧数组
- 写操作开销大,因为要复制整个数组; 读操作完全无锁, 直接读数组就行,所以它适合**读多写少**场景,比如配置列表,监听器这种写入频率低但读取频繁的场景
- 写时复制机制: 写操作加锁; 复制当前数组得到新数组; 在新数组上修改; 用新数组替换旧数组; 读操作无锁,直接读当前数组

Java 的 CopyOnWriteArrayList 和 Collections.
synchronizedList 有什么区别？分别有什么优缺点？
- 两者都能让List线程安全,但实现思路不同;CopyOnWriteArrayList靠写时复制,synchronizedList靠加锁
- CopyOnWriteArrayList每次写入都会复制整个底层数组,生成一个新数组来承载修改,读操作直接读旧数组,完全不用加锁;synchronizedList就是给原来的List套一层壳,每个方法都加synchronized,读写要抢同一把锁
- 读多写少用CopyOnwriteArrayList,写也不少就synchronizdList或干脆上ConcurrentLinkedQueue

你遇到过 ConcurrentModificationException 错误吗？它是如何产生的？
- 这是Java集合中的fail-fast机制抛出的异常,专门用来检测 "一边遍历一遍改" 的操作
- 在非线程安全的集合里,如果你边迭代边修改,可能会出现数据丢失,跳过元素等奇怪问题,Java不想让你稀里糊涂跑下去,干脆检测到修改直接抛异常,让问题早点暴露出来
- 内部实现: 集合内部维护一个modCount遍历,记录修改次数,每次add,remove都会让modCount加1. 迭代器创建时会把当前modCount存到expectedModCount里,每次调用next()前都检查这俩值是否相等,不相等就抛ConcurrentModificationException

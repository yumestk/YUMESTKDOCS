⚙️ 一、 **原理篇：四层递进结构（必背骨架）回答原理时，直接甩出这四个层级，层次感瞬间拉满：**

对象头层面： 锁状态存在于 Java 对象的 Mark Word（对象头）中。
字节码层面： 编译后对应 monitorenter 和 monitorexit 两条指令。
JVM 逻辑层面（Monitor 监视器）： 每个锁对象都关联一个 Monitor。Owner： 抢到锁的线程。
EntryList： 没抢到锁、处于阻塞状态的线程队列。
流程： 执行 monitorenter 时，若 Owner 为空则占有；若被占用则进 EntryList。
操作系统层面： 传统的重量级锁是向 OS 申请 Mutex（互斥量），涉及用户态和内核态的切换，开销很大。

📈 二、 **锁升级：三大竞争场景（故事化记忆）不要死记硬背每个锁的细节，把锁升级记忆为“为了适配不同竞争强度，逐步升级”的过程。**

场景 1：完全没有竞争，只有一个线程（无锁 $\rightarrow$ 偏向锁）核心： 对象的 Mark Word 直接记录该线程 ID。效果： 这个线程下次再来（重入），直接对比 ID 就能拿锁，完全没有额外开销。

场景 2：有轻微竞争，两三个线程交替执行（偏向锁 $\rightarrow$ 轻量级锁）升级时机： 来了新线程争抢，偏向锁被撤销，升级为轻量级锁。核心： 线程在自己的栈帧里创建锁记录，通过 CAS 自旋去修改对象头指针。效果： 避免了向操作系统申请 Mutex，用 CPU 自旋（空转）换取了线程不被阻塞。

场景 3：竞争激烈，大量线程围剿（轻量级锁 $\rightarrow$ 重量级锁）升级时机： 自旋次数过多，或者多个线程同时轰炸。核心： 发生锁膨胀，升级为重量级锁。效果： 未抢到锁的线程不再自旋（太耗 CPU），而是直接进入 Monitor 的 EntryList 挂起阻塞，等待唤醒。

🎯 **话术总结（面试脱口而出版）面试官： 讲一下 Synchronized 的底层原理和锁升级过程。**

你： “Synchronized 经过 JDK 1.6 的优化后，它的设计思路是适配不同竞争强度下的场景，以空间和 CPU 换性能。
在 JVM 底层，它通过 Monitor（监视器） 来管理线程，对应字节码的 monitorenter 和 monitorexit。它的锁状态记录在对象的 Mark Word 里。

整个锁升级是不可逆的，分为三个阶段：
单线程时是偏向锁，只记录线程 ID，重入开销极低；

出现轻微竞争时升级为轻量级锁，线程通过 CAS 自旋去抢锁，避免了系统调用；

当竞争激烈自旋无果时，膨胀为重量级锁，底层向操作系统申请 Mutex 互斥量，没抢到锁的线程进入 EntryList 队列挂起阻塞，防止跑满 CPU。

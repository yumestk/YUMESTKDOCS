## 📌 项目一句话定位
这是一个面向“有页面/后台搭建需求但研发资源有限”的业务团队的 AI 零代码平台：用户通过自然语言驱动代码生成、预览、定点修改与部署，核心价值是把“AI demo 生成”升级为“可治理、可追溯、可运营的工程化生产链路”。

## 🧱 层级拆解

### 功能层
- 自然语言创建应用，自动路由生成类型（HTML / MULTI_FILE / VUE_PROJECT），并流式返回生成过程（`src/main/java/com/yume/yuaicodemother/controller/AppController.java:60`）。
- 对话页内实时预览、可视化选中元素后继续对话修改，形成“看见即改”闭环（`yume-ai-code-mother-frontend/src/pages/app/AppChatPage.vue:435`）。
- 支持应用部署、代码下载、应用信息编辑与管理端运营能力（精选、优先级、用户/应用/对话管理）。

### 系统层
- **后端分层**：Controller（REST/SSE）→ Service（业务编排）→ Core（生成外观/流处理/构建）→ AI（LangChain4j + Tool + Memory）→ Workflow（LangGraph4j）（`src/main/java/com/yume/yuaicodemother/service/impl/AppServiceImpl.java:78`）。
- **AI 编排链路**：`image_collector -> prompt_enhancer -> router -> code_generator -> code_quality_check -> (build|skip|regenerate)`，通过条件边+循环边做质检回路（`src/main/java/com/yume/yuaicodemother/langgraph4j/CodeGenWorkflow.java:51`）。
- **接口契约**：生成接口统一 SSE，普通 chunk 和业务错误都用事件化协议返回，前端可区分 `done` / `business-error`（后端：`src/main/java/com/yume/yuaicodemother/exception/GlobalExceptionHandler.java:76`，前端：`yume-ai-code-mother-frontend/src/pages/app/AppChatPage.vue:539`）。
- **记忆机制**：按 `appId` 隔离对话上下文，Redis 存储 + DB 历史回放，避免模型“失忆”与重启丢上下文（`src/main/java/com/yume/yuaicodemother/ai/AiCodeGeneratorServiceFactory.java:88`，`src/main/java/com/yume/yuaicodemother/service/impl/ChatHistoryServiceImpl.java:101`）。

### 工程层（含待补强问题）
已体现：
- 权限：`@AuthCheck + AOP` 控制管理员接口（`src/main/java/com/yume/yuaicodemother/aop/AuthInterceptor.java:33`）。
- 限流：Redisson 分布式令牌桶，支持 USER/API/IP 维度（`src/main/java/com/yume/yuaicodemother/ratelimter/aspect/RateLimitAspect.java:37`）。
- 缓存：Caffeine 缓存 AI 服务实例 + Redis Cache 缓存精选分页（`src/main/java/com/yume/yuaicodemother/ai/AiCodeGeneratorServiceFactory.java:52`，`src/main/java/com/yume/yuaicodemother/config/RedisCacheManagerConfig.java:43`）。
- 超时：Vue 构建命令设置超时并强制终止（`src/main/java/com/yume/yuaicodemother/core/builder/VueProjectBuilder.java:124`）。

## 🧠 核心模块面试讲法

### 模块1：AI 代码生成主链路（SSE + 工作流）
- **痛点**  
  传统“单次 prompt 生成”不可控，用户看不到中间过程，失败时无法定位是模型问题还是工程问题。
- **指标**  
  - 可用性：成功生成可预览代码的比例  
  - 体验：首 token 时间、完整生成耗时  
  - 质量：质检通过率、回路重生成次数
- **选型对比**  
  - 选 LangGraph4j 而不是“单 Agent 串行调用”：因为需要显式条件路由和质检回环。  
  - 选 SSE 而不是轮询：低开销实时反馈，用户不必等待整包结果。
- **架构链路**  
  `AppController` 接收请求 → `AppServiceImpl` 校验权限并落库用户消息 → `AiCodeGeneratorFacade` 按类型调用 AI → `StreamHandlerExecutor` 分类处理流并保存 AI 消息 → SSE 回前端渲染（`src/main/java/com/yume/yuaicodemother/controller/AppController.java:60`, `src/main/java/com/yume/yuaicodemother/service/impl/AppServiceImpl.java:95`, `src/main/java/com/yume/yuaicodemother/core/AiCodeGeneratorFacade.java:77`）。
- **验证方法**  
  - 用生成成功率、平均重试回路次数评估工作流有效性。  
  - 对比“有质检回路/无回路”的最终可构建率。
- **运维考虑**  
  - 流式错误统一转 `business-error` 事件，前端可感知并收敛状态。  
  - 用户维度限流防止刷爆模型调用（`src/main/java/com/yume/yuaicodemother/ratelimter/aspect/RateLimitAspect.java:64`）。

### 模块2：多轮记忆与会话一致性
- **痛点**  
  多轮迭代中模型容易丢上下文，导致“上一轮改过的内容又改回去”。
- **指标**  
  - 连续多轮改动的一致性命中率  
  - 二次描述率（用户是否频繁重复背景）
- **选型对比**  
  - 仅 DB 历史：可追溯但推理时检索/拼装成本高。  
  - 仅内存：低延迟但服务重启丢失。  
  - 当前 Redis ChatMemory + DB 回放：兼顾实时与持久。
- **架构链路**  
  工厂按 `appId + codeGenType` 取/建 AI 服务实例，初始化时从 DB 载入最近消息到 Redis-backed memory（`src/main/java/com/yume/yuaicodemother/ai/AiCodeGeneratorServiceFactory.java:75`, `src/main/java/com/yume/yuaicodemother/service/impl/ChatHistoryServiceImpl.java:101`）。
- **验证方法**  
  - 构造 5~10 轮连续修改 case，统计遗漏历史约束的比例。  
  - 重启服务后回放验证上下文恢复效果。
- **运维考虑**  
  - MessageWindow 上限控制 token 成本（当前 maxMessages=20）。  
  - 需要补充：不同用户隔离与隐私脱敏审计策略。

### 模块3：可视化定点编辑（预览 + 元素上下文注入）
- **痛点**  
  纯文本描述修改页面成本高、定位不准（“把这个按钮改大一点”中的“这个”不明确）。
- **指标**  
  - 单轮修改命中率  
  - 用户补充描述次数  
  - 从发起修改到预览更新耗时
- **选型对比**  
  - 纯聊天：实现简单，但语义歧义高。  
  - 可视化选中 + 上下文注入：增加前端复杂度，但显著降低歧义。
- **架构链路**  
  前端 iframe 选中元素，提取 selector/text/path 拼进 prompt，再走统一 SSE 生成链路（`yume-ai-code-mother-frontend/src/pages/app/AppChatPage.vue:436`）。
- **验证方法**  
  - 固定改版任务集，对比“有无元素上下文”一次成功率。  
  - 人工抽样核对是否改到指定 DOM。
- **运维考虑**  
  - 需补充：跨域 iframe 场景的安全策略与降级方案。  
  - 需补充：错误注入（selector 无效、页面重绘）下的恢复路径。

### 模块4：工程治理（权限、限流、缓存、构建部署）
- **痛点**  
  AI 应用若缺少治理，容易出现越权、资源打满、响应抖动、部署失败不可控。
- **指标**  
  - 权限误放行率=0  
  - 限流拦截率与误杀率  
  - 缓存命中率与接口 P95  
  - 构建成功率与超时率
- **选型对比**  
  - 限流：本地内存限流 vs Redisson 分布式限流；后者适合多实例。  
  - 缓存：仅 Redis vs Caffeine+Redis；后者降低热点访问延迟。
- **架构链路**  
  `@AuthCheck` AOP 校验角色（`src/main/java/com/yume/yuaicodemother/aop/AuthInterceptor.java:33`）；`@RateLimit` 切面统一限流（`src/main/java/com/yume/yuaicodemother/ratelimter/aspect/RateLimitAspect.java:37`）；`@Cacheable` 缓存精选列表（`src/main/java/com/yume/yuaicodemother/controller/AppController.java:267`）；部署前对 Vue 项目强制 build 且超时控制（`src/main/java/com/yume/yuaicodemother/core/builder/VueProjectBuilder.java:75`）。
- **验证方法**  
  - 压测下验证限流生效与服务稳定性。  
  - 构建失败场景注入（依赖下载慢、构建脚本异常）验证错误可观测。
- **运维考虑**  
  - 建议补充告警：限流突增、SSE断流率、部署失败率。  
  - 建议补充回滚：部署目录版本化 + 快速回切机制。

## ⚔️ 技术选型横向对比题库

1) **为什么你用 LangGraph4j，不直接一个 LangChain4j Agent 跑到底？**  
回答要点：
- 我的问题不是“能不能生成”，而是“能否工程可控”。
- 需要显式表达条件边（质检通过/失败）和循环边（失败回退重生成），Graph 对状态机更友好。
- 这样上线后可做节点级观测和故障定位，而不是黑盒 prompt。

2) **为什么用 SSE，不用 WebSocket？**  
回答要点：
- 场景是服务端单向流式输出 token，SSE 足够且接入成本低。
- HTTP 语义友好，和现有网关/鉴权链路兼容性更好。
- 若未来要做双向协同编辑、实时协作，再评估 WebSocket。

3) **为什么是 Caffeine + Redis 双层缓存，不只用 Redis？**  
回答要点：
- AI 服务实例/高频对象本地命中能显著降低延迟和序列化开销。
- Redis 负责跨实例共享与一致性边界。
- 权衡是本地缓存带来短暂不一致风险，所以只放“可容忍短暂不一致”的读多场景。

4) **如果流量放大 10 倍，这套方案先炸在哪？怎么扛？**  
回答要点：
- 首先是模型调用并发、SSE连接数、构建任务（npm build）资源占用。
- 扩容策略：生成与构建异步解耦、构建任务队列化、实例水平扩展+用户限流分层。
- 成本策略：路由轻量模型处理低复杂需求，重模型只给高复杂任务。

## 🎤 面试结尾 30 秒讲法（可直接用）
这个项目我重点不是“接了个模型 API”，而是把生成链路做成了可治理系统：前面有路由与增强，中间是可回路的工作流，后面有流式交互、权限限流、缓存和部署闭环。若继续工程化，我下一步会补齐评测体系和可观测指标，让它从“能用”变成“可规模化运营”。
